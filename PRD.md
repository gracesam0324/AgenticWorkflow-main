> **문서 상태**: Pre-implementation PRD (Claude Code workflow.md 제작 전 방향 설정 문서)
> **기반**: 15개 AI teammate 에이전트 심층 리서치 (Phase 1-3 완료)
> **시나리오**: Balanced + Graduated Autonomy 차별점
> **작성일**: 2026-04-03

이 PRD는 **Claude Code workflow.md 제작을 위한 사전 문서**다. 구현 세부사항보다 *무엇을 왜 만드는가*에 집중한다. 모든 결정은 아래 절대 기준 순서로 평가한다.

## 1. 제품 정체성

### 한 문장 정의

> **"사용자가 만들고 싶은 것을 말하면, 로컬 컴퓨터에서 자동으로 구현해주는 신뢰할 수 있는 AI 에이전트 시스템"**

### 이것은 무엇이 아닌가

| ❌ 아닌 것 | ✅ 맞는 것 |
|---|---|
| SaaS 서비스 | 로컬 컴퓨터에서 완전히 실행 |
| 코드 생성 도구 | 생성 + 배포 + 실행 + 유지보수 |
| AI 챗봇 | 실제로 작동하는 자동화 실행기 |
| "빌더 도구" | **신뢰성 인프라** |

### 핵심 고객 문제

**마커스의 문제**: "자동화를 *못 만드는* 게 아니다. 기존 자동화의 **유지보수 비용**이 너무 크다."

> 마커스는 n8n, Python 스크립트, 셸 명령어로 자동화를 만들 수 있다. 하지만 그것들이 깨질 때마다 복구하는 시간이 새로 만드는 시간보다 많다. 이 제품은 그 유지보수 비용을 제거한다.

**사라의 문제**: "자동화를 만들 줄 모른다. 그러나 매주 월요일 아침 45분을 반복 작업에 쓰고 있다."

> 사라는 터미널을 열지 않는다. 설정이 복잡하면 탭을 닫는다. 이 제품이 그녀를 위해 존재하려면, 첫 30분 안에 성공 경험을 만들어야 한다.

## 2. 절대 기준

> 모든 기능 결정, 아키텍처 결정, 출시 결정은 이 순서로 평가한다.

### 절대 기준 1: 안전 우선 — 코드 레벨 경계

**`IProcessExecutor`는 이 시스템의 면역 체계다.**

- 모든 셸 실행, 파일 작업, 프로세스 생성은 이 인터페이스를 통과한다
- 예외 없음. LLM 프롬프트로 이 경계를 우회할 수 없다
- 프롬프트 엔지니어링은 안전 경계가 아니다 — 코드만이 안전 경계다
- 이 인터페이스는 Week 1 첫 커밋에서 추출된다. 기능 개발 전에.

**근거**: 단 하나의 "이 AI가 내 파일을 삭제했다" Reddit 포스트가 제품을 끝낼 수 있다.

### 절대 기준 2: 로컬 퍼스트 — 타협 없음

- 모든 실행은 사용자 로컬 머신에서 일어난다
- 자격증명, 워크플로우 데이터, 실행 로그는 기기를 떠나지 않는다
- 클라우드 동기화 옵션은 없다 (요청이 와도 v1에서는 추가하지 않는다)
- DPAPI SecretStoreService (이미 구현됨)가 모든 비밀 데이터를 처리한다

**근거**: 이 제품을 선택하는 이유 자체가 "내 데이터가 내 컴퓨터에 있어야 한다"는 것이다. 이 정체성을 희석하면 Zapier나 n8n과 구별되지 않는다.

### 절대 기준 3: 불변 플랜 — 승인 후 수정 불가

- 사용자가 워크플로우 플랜을 승인하면, 그 플랜은 잠긴다
- 실행 중 LLM이 플랜을 재구성하는 것은 금지된다
- 변경이 필요하면 사용자가 새 플랜을 만들고 다시 승인해야 한다
- 체크섬으로 플랜 불변성을 코드 레벨에서 강제한다

**근거**: "자율 에이전트가 실행 중에 계획을 바꾼다"는 것이 자동화 공포의 핵심이다. 이를 제거하면 신뢰가 시작된다.

### 절대 기준 간 충돌 시

안전 > 로컬 퍼스트 > 불변 플랜 > 기능 완성도

## 3. 사용자 정의

### 페르소나 1: 마커스 (1차 타겟 — 배포 채널)

| 항목 | 내용 |
|---|---|
| **역할** | 솔로 기술 창업자 / 1-3인 팀 개발자 |
| **문제** | 자동화 유지보수 비용이 새로 만드는 시간을 초과 |
| **현재 도구** | n8n, Python 스크립트, 셸 명령어 (믿을 수 없음) |
| **비타협 요구사항** | 자격증명 로컬 저장, 자가치유, 완전 배포 |
| **제품과의 관계** | 직접 배포, 이 제품 없으면 매일 4시간 낭비 |

**마커스가 쓰는 이유**: 이 제품이 없으면 그는 n8n이 새벽 2시에 깨지는 것을 고치러 SSH를 연다. 이 제품이 있으면 자가치유가 처리하고 아침에 평어체 보고서를 받는다.

**마커스가 쓰지 않는 이유 (회피해야 할 것)**:
- 자격증명이 클라우드 서버를 통과한다
- 워크플로우가 조용히 실패한다 (자가치유 없음)
- 코드를 생성하지만 실행/스케줄링은 직접 해야 한다

### 페르소나 2: 사라 (2차 타겟 — 수익 타겟)

| 항목 | 내용 |
|---|---|
| **역할** | 마케팅 운영 매니저 / 비기술직 지식 근로자 |
| **문제** | 매주 반복 작업 (보고서, 데이터 수집) 45분 소비 |
| **현재 도구** | Zapier (비용 때문에 불만), Excel 수동 작업 |
| **비타협 요구사항** | 터미널 없음, 첫 30분 성공, 감사 로그 |
| **제품과의 관계** | 마커스가 소개해줌. 한 번 실패하면 돌아오지 않음 |

**중요**: 사라를 위해 *설계*하되, 마커스를 통해 *배포*한다. 사라에게 직접 마케팅하는 것은 v2.

### 타겟팅 시퀀스 (순서가 중요)

```
Month 1-2: 마커스 타입 20-50명 (초대 전용)
           → 역할: 안정성 검증 + 배포 채널

Month 3-4: 200-500명 확장 베타 (IT 관리자, 데이터 분석가)
           → 전제: Month 3 게이트 통과 후

Month 5+:  공개 출시 (사라 타입 포함 가능)
           → 전제: 베타 데이터 충분히 축적 후
```

## 4. 핵심 차별점: Graduated Autonomy with Audit Trail

> 이것이 n8n, AutoGPT, Claude Code와 다른 유일한 이유다.

### 작동 방식

**레벨 시스템 (3단계)**:
- `Paranoid` (기본): 모든 단계 전 명시적 승인 필요
- `Balanced`: 비파괴적 작업은 자동, 파괴적 작업만 승인
- `Trust`: 명백히 위험한 작업만 승인

**자동 레벨 상승**:
- 10번 성공 후: "이 레벨에서 4번의 불필요한 승인이 있었습니다. Balanced로 올리시겠습니까?"
- 25번 성공 후: Balanced → Trust 제안
- 레벨 상승은 **사용자가 결정**한다. 시스템이 결정하지 않는다.

**레벨 자동 하강**:
- 높은 레벨에서 실패 발생 시 → 이전 레벨로 자동 복귀 + 이유 설명
- 신뢰는 증명으로 얻는다. 가정하지 않는다.

**실행 후 내러티브**:
```
[실행 완료] 7개 단계 실행됨.
- 3단계: 승인 필요했음
- 5단계: 실패 후 자동 복구됨
- "Balanced 레벨이었다면 승인 2회가 생략되었을 것입니다"
```

### 왜 이것이 중요한가

- **Aggressive의 문제**: 처음부터 너무 많은 자율성 → 신뢰가 쌓이기 전에 사고 발생
- **Conservative의 문제**: 항상 모든 것을 승인 → 자동화의 가치 훼손
- **이 제품**: 신뢰는 사용 데이터를 통해 단계적으로 확장됨

## 6. 기술 아키텍처 원칙

### 인터페이스 추출: 딱 3개

```csharp
// 1. IProcessExecutor — 안전 경계 + 테스트 가능성
interface IProcessExecutor {
    ExecutionResult Execute(Command command);
    SafetyLevel Classify(Command command);
    // WindowsProcessExecutor (v1)
    // DryRunProcessExecutor (항상 존재)
    // MacProcessExecutor (Yellow Zone)
}

// 2. ILLMProvider — 공급자 유연성
interface ILLMProvider {
    Task<LLMResponse> CompleteAsync(LLMRequest request);
    ModelCapabilityTier Tier { get; }
    // AnthropicProvider (v1 기본)
    // OpenAICompatibleProvider (v1)
    // OllamaProvider (Yellow Zone)
}

// 3. IWorkflowPlanStore — 불변성 계약 강제
interface IWorkflowPlanStore {
    void Save(WorkflowPlan plan); // 한 번만 쓰기
    WorkflowPlan Load(string id); // 체크섬 검증 포함
    // 수정은 새 플랜 생성으로만 가능
}
```

**이 3개 이외에는 추출하지 않는다.** 추출 요구가 생길 때 추출한다.

### 코드베이스 전략

기존 cmux-windows 코드베이스 위에 **확장**한다. 재작성하지 않는다.

```
재사용 (수정 없이):
  - SecretStoreService (DPAPI) ✅
  - NamedPipeServer IPC ✅
  - SessionPersistenceService (원자적 쓰기 패턴) ✅
  - AgentConversationStoreService ✅
  - ConPTY TerminalSession ✅

확장 (새 인터페이스 추가):
  - AgentRuntimeService → ILLMProvider 추출
  - bash_run → IProcessExecutor 추출

신규 구현:
  - WorkflowPlanEngine (NL → YAML)
  - WorkflowExecutionEngine (단계별 실행)
  - GraduatedAutonomyTracker (레벨 관리)
  - AuditLogService (append-only)
  - SelfHealingEngine (유료)
```

### Last Mile A / B 분리

```
Last Mile A (v1 구현):
  자연어 → 실행 → 결과 보고 (일회성)
  "NL to fully executed, one-time automation"

Last Mile B (v1.1 구현, 스케줄링 Yellow Zone 조건 충족 후):
  스케줄링 + 배경 실행 + 재부팅 후에도 지속
  조건: 감사 로그 60일 운영 + 실패 보고 검증 완료
```

### 플랫폼

- **v1**: Windows 전용 (WPF UI + ConPTY)
- **워크플로우 엔진**: 플랫폼 중립적 (`cmux.WorkflowEngine` 별도 프로젝트)
- **macOS**: Yellow Zone (IProcessExecutor macOS 구현, UI는 별도)
- **Linux**: Red Zone (개발자는 이미 자체 도구를 만들 수 있음)

> 5개 조건 중 하나라도 미달이면 유료 티어를 출시하지 않는다.

```
게이트 조건 (모두 충족 필수):

  ① Day-30 Active Workflow Rate ≥ 40%
     정의: 설치 후 30일 시점에도 워크플로우를 실행한 사용자 비율

  ② 자발적 지불 의향 10건 이상
     정의: 설문이 아닌 대화에서 "이거 돈 내고 쓰겠다" 발언

  ③ 치명적 실패 ZERO
     정의: 데이터 손실, 시스템 손상, 승인 없는 파괴적 작업
     (1건 발생 시 클록 리셋 — 이후 30일 무결 필요)

  ④ 자가치유 복구율 ≥ 30%
     정의: 워크플로우 실패 중 자동 복구된 비율

  ⑤ 큐레이션 템플릿 9/10 성공률
     정의: 팀이 설정하지 않은 깨끗한 머신 5대에서 각 템플릿 실행
```

**게이트 실패 시 대응**:
1. 확장하지 않는다
2. 무엇이 왜 실패했는지 조사한다
3. 60일 후 재평가한다
4. 게이트 기준을 낮추는 것은 금지한다

## 9. 주요 지표 (KPI)

### 1차 지표 (이것만 보면 됨)

> **Day-30 Active Workflow Rate (AWR)**
> 정의: 설치 후 30일 시점에도 워크플로우를 실행한 사용자 비율
> 목표: 40%+ (게이트), 55%+ (6개월 말)

### 2차 지표

| 지표 | 목표 | 의미 |
|---|---|---|
| 자가치유 복구율 | 30%+ (게이트), 45%+ (6개월) | 신뢰성 |
| 첫 성공까지 시간 | 8분 이내 | 온보딩 품질 |
| 실패 보고 명확도 | 90%+ 사용자가 복구 행동 이해 | 실패 UX 품질 |
| NPS | 50+ (알파), 60+ (베타) | 추천 의향 |
| 템플릿 성공률 | 9/10 (깨끗한 머신) | 제품 신뢰성 |

### 금지 지표 (이것에 최적화하면 안 됨)

- GitHub 스타 수 (AutoGPT: 160K 스타, <5K 유료 고객)
- 총 설치 횟수
- 페이지뷰 / DAU 단순 합계

## 10. 배포 전략

### 단계별 배포 시퀀스

```
Stage 1: Private Alpha (Month 1-2)
  대상: 마커스 타입 20-50명, 개인 초대
  채널: 팀 네트워크, GitHub 기존 사용자
  목표: 9/10 성공률 검증, NPS 50+

Stage 2: Closed Beta (Month 3-4)
  전제: Month 3 게이트 통과
  대상: 200-500명 (기술 관리자, 데이터 분석가 포함)
  채널: Typeform 대기자 + Alpha 사용자 추천
  목표: Day-30 AWR 40%+

Stage 3: Public Launch (Month 5+)
  전제: Closed Beta 데이터 충분
  채널: HackerNews Show HN, winget 등록
  주의: ProductHunt는 공개 데이터 축적 후에만

Stage 4: Amplification (Month 7+)
  전제: MRR 양의 성장, NPS 60+, 실제 사례 3건
  채널: ProductHunt, 개발자 뉴스레터, 콘텐츠 마케팅
```

### 배포 원칙

- 공개 채널은 신호를 10,000배 증폭한다
- 신호가 긍정적이면 증폭은 자산이다
- 신호가 부정적이면 증폭은 치명적이다
- Stage 3 이전에 공개 채널을 사용하지 않는다

## 11. 명시적 제약 ("절대 하지 않을 것")

이 목록은 기능 제외보다 강력하다. 팀 내에서 논의가 생기면 이 문서를 참조한다.

```
1. 클라우드로 데이터 전송 (옵션이어도 안 됨)
   → 로컬 퍼스트 정체성은 타협 불가

2. LLM으로 안전 결정
   → 안전 경계는 항상 결정론적 코드

3. Month 3 게이트 기준 낮추기
   → 게이트는 타협 불가, 연장만 가능

4. 스케줄링을 v1에 포함
   → 보이지 않는 실패는 보이는 실패보다 위험

5. 커뮤니티 템플릿 공유 (v1)
   → 품질 제어 없는 공유는 RCE 벡터

6. GitHub 스타를 성공 지표로 사용
   → AWR이 진짜 지표

7. 마커스와 사라를 위한 동일한 UI
   → 둘 다 실패함. 마커스 UI 먼저, 사라 레이어는 그 위에

8. 게이트 통과 전 유료화
   → 실망한 유료 사용자는 실망한 무료 사용자보다 나쁨
```
