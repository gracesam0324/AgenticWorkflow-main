## 1. 설계 철학

### 1.1 핵심 신념: 자서전은 공학이자 예술이다

이 시스템의 근본적 신념:

> **자서전 생성은 "기억을 텍스트로 변환하는 작업"이 아니라, "한 인간의 삶에서 의미를 발견하고 서사로 구성하는 작업"이다.**

이 신념이 모든 설계 결정을 관통한다:

| 관점 | 단순 변환 접근 | 이 시스템의 접근 | 설계 근거 |
|------|-------------|---------------|----------|
| 인터뷰 | 질의응답 → 텍스트 | 라포 형성 → 감각적 디테일 추출 | 좋은 산문의 원재료는 감각이다 |
| 구조화 | 시간순 나열 | 킥포인트 기반 서사 아키텍처 | 인생은 날짜가 아니라 전환점으로 기억된다 |
| 글쓰기 | 사실 기록 | 보이스 블렌딩 + 문학적 산문 | 자서전은 보고서가 아니라 문학이다 |
| 검증 | 맞춤법 검사 | 7차원 적대적 리뷰 + CoVe | 품질은 자동화된 비판에서 나온다 |

### 1.2 품질 절대주의의 자서전 도메인 적용

부모 게놈에서 유전된 **절대 기준 1(품질 우선)**이 자서전 도메인에서 발현되는 방식:

> **유일한 성공 측정 기준은 "읽을 가치가 있는 책"이다.**
> 속도, 토큰 비용, 분량 목표는 산문 품질, 사실 정확성, 감정적 진정성보다 부차적이다.

구체적 행동 규칙:

| 상황 | 일반적 선택 | 이 시스템의 선택 | 근거 |
|------|-----------|---------------|------|
| 인터뷰 세션 수 | 3회 고정 | 커버리지가 충분할 때까지 (최소 3회) | 빈 인생 시기가 있으면 해당 챕터 품질 저하 |
| 챕터 수정 횟수 | 1회 패스 | 리뷰어가 PASS할 때까지 (최대 3-5라운드) | 첫 드래프트가 최종본이 되어서는 안 된다 |
| 교차 챕터 일관성 | 선택적 검토 | 전체 원고 8차원 감사 (CC1-CC8) | 챕터 5의 인물이 챕터 8에서 다른 이름으로 등장하면 독자 신뢰가 무너진다 |
| 보이스 일관성 | 주관적 판단 | Python 계산 메트릭 + 허용 오차 | "느낌"이 아니라 "수치"로 일관성을 측정해야 재현 가능하다 |
| 스타일 선택 | AI가 자동 결정 | 사용자와 3단계 캘리브레이션 | 화자의 목소리는 화자가 결정해야 한다 |

### 1.3 Generator-Critic 패턴의 이중 적용

이 시스템의 아키텍처적 핵심은 **Generator-Critic 패턴**이 두 수준에서 작동하는 것이다:

```
Level 1: 챕터 수준
  @chapter-writer (Generator) ←→ @reviewer (Critic)
  각 챕터가 리뷰를 통과할 때까지 수정 순환

Level 2: 원고 수준
  전체 챕터 (Generator output) ←→ @reviewer + @reviewer-deep (Critics)
  교차 챕터 일관성 감사로 전체 원고의 품질 보장
```

이 이중 구조가 필요한 이유:
- **Level 1만 있으면**: 개별 챕터는 우수하지만 챕터 간 모순이 발생한다
- **Level 2만 있으면**: 전체 일관성은 있지만 개별 챕터의 산문 품질이 불균일하다
- **둘 다 있어야**: 부분의 품질(산문) × 전체의 품질(일관성) = 출판 품질

## 2. 왜 AI 자서전인가

### 2.1 자서전 작성의 전통적 과정과 병목

전통적 자서전 작성 과정:

```
1. 주제 인물 인터뷰 (수십 시간)
2. 녹취 정리 (인터뷰 시간의 3-5배)
3. 서사 구조 설계 (수주)
4. 초고 집필 (수개월)
5. 편집·교정 (수주-수개월)
6. 조판·출판 (수주)
```

**병목 분석**:

| 단계 | 병목 원인 | AI가 해소하는 방법 |
|------|---------|-----------------|
| 인터뷰 | 전문 인터뷰어 섭외 비용 | `@interviewer`가 감각적 디테일을 추출하는 체계적 질문 |
| 녹취 정리 | 반복적 수작업 | JSON 스키마로 구조화된 인터뷰 기록 |
| 서사 구조 | 한 명의 편집자 역량 의존 | `@story-architect`가 데이터 기반으로 킥포인트·테마·아크 분석 |
| 초고 집필 | 고스트라이터 비용·시간 | `@chapter-writer`가 보이스 가이드 기반으로 산문 생성 |
| 편집 | 주관적 판단 편차 | `@reviewer`의 7차원 정량 평가 + Python 결정론적 검증 |
| 조판 | 전문 기술 필요 | Pandoc + XeLaTeX 자동 빌드 파이프라인 |

### 2.2 AI가 잘하는 것과 못하는 것의 경계

이 시스템의 설계는 AI의 강점과 약점을 명확히 인식한다:

| AI가 잘하는 것 | AI가 못하는 것 | 설계 대응 |
|--------------|-------------|----------|
| 대량 텍스트에서 패턴 추출 | 실제 기억의 진위 판별 | 인터뷰를 인간과의 실시간 대화로 진행 |
| 일관된 문체 유지 | 화자의 고유한 뉘앙스 창조 | 보이스 지문 → 수치 메트릭 → 결정론적 가이드 |
| 구조적 일관성 검증 | 감정의 진정성 판단 | `@reviewer`의 LLM 기반 질적 평가 (7차원) |
| 대규모 텍스트 교차 참조 | "읽을 때의 감동" 측정 | Human Gate 4곳 (Step 3, 4, 7, 11) |
| 반복적 수정 작업 | 서사적 위험 감수 | `@story-architect`가 3개 후보 구조 제안 → 사용자 선택 |

**핵심 원칙**: AI가 잘 못하는 영역은 **사람에게 위임**(Human Gate)하거나, **결정론적 코드로 대체**(Python 검증)한다. AI에게 맡기는 영역은 **적대적 리뷰로 견제**한다.

## 3. 파이프라인 아키텍처

### 3.1 12단계 파이프라인의 설계 원리

```
┌─────────────────────────────────────────────────────────────────────┐
│                    RESEARCH (Steps 1-4)                              │
│  "재료 수집" — 인터뷰에서 원재료를 확보하고 서사 설계도를 합의한다        │
├─────────────────────────────────────────────────────────────────────┤
│  Step 1: Interview Planning                                         │
│    Orchestrator → interview_plan.md                                 │
│                                                                     │
│  Step 2: Conduct Interviews (반복)                                   │
│    @interviewer → INT-{NNN}.json (세션별)                            │
│    ✓ 최소 3세션, 커버리지 충분할 때까지                                │
│    ✓ 매 세션 후 사용자와 3개 질문 프로토콜                             │
│                                                                     │
│  Step 3: (human) Interview Review                                   │
│    사용자가 인터뷰 완성도 검토 → 승인/추가 세션                         │
│                                                                     │
│  Step 4: (human) Story Blueprint Co-Creation                        │
│    @story-architect + 사용자 → story_blueprint.json                  │
│    ✓ 3개 후보 구조(시간순/테마/킥포인트) 제안                          │
│    ✓ 사용자의 핵심 메시지, 전환점, 강조/배제 영역 반영                  │
│    ✓ 한-흥-정 감정 아크 설계                                          │
└────────────────────────────────┬────────────────────────────────────┘
                                 │ voice_fingerprint.json (Python 계산)
                                 │ story_blueprint.json (사용자 합의)
                                 ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    PLANNING (Steps 5-8)                              │
│  "설계 확정" — 스토리 바이블을 구축하고 글쓰기 스타일을 확정한다         │
├─────────────────────────────────────────────────────────────────────┤
│  Step 5: Story Bible Construction                                   │
│    @story-architect → story_bible.json                              │
│    ✓ 캐릭터 레지스트리 (정규 이름, 별칭, 관계 아크)                    │
│    ✓ 크로노로지컬 타임라인 (날짜/시기별 정렬)                          │
│    ✓ 장소 레지스트리 (감각적 디테일 포함)                               │
│    ✓ 테마 분석 (5-10개 유기적 테마)                                    │
│    ✓ 챕터 플랜 (블루프린트 구속)                                       │
│    ✓ 보이스 가이드 (Python 메트릭 기반)                                │
│    ✓ 팩트 레지스트리 (검증 가능한 사실)                                 │
│                                                                     │
│  Step 6: Story Bible Review                                         │
│    @reviewer → RV-story-bible.json                                  │
│    ✓ CoVe 프로토콜 (클레임 추출 → 인터뷰 대조 검증)                    │
│    ✓ 블루프린트 준수 여부 검증                                         │
│                                                                     │
│  Step 7: (human) Story Bible Approval                               │
│    사용자가 스토리 바이블 + 리뷰어 피드백 검토 → 승인/수정               │
│                                                                     │
│  Step 8: (human) Writing Style Selection & Calibration              │
│    10종 스타일 갤러리 → 사용자 선택 → 3개 캘리브레이션 샘플              │

│    ✓ FAIL 시 9b로 복귀 (최대 3-5라운드)                               │

│  Step 10: Cross-Chapter Consistency Check                           │
│    @reviewer + @reviewer-deep → consistency-report.json             │

│    ✓ 치명적 불일치 → 해당 챕터 9b로 복귀                               │
│                                                                     │
│  Step 11: (human) Manuscript Review                                 │
│    사용자가 전체 원고 최종 검토 → 승인/수정                              │
│                                                                     │
│  Step 12: Book Build                                                │
│    Pandoc + XeLaTeX → PDF(EN/KO) + EPUB(EN/KO)                     │
│    ✓ 전문·후문 포함 (제목 페이지, 헌사, 서문, 감사의 글)                │

└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 단계 간 의존성과 게이트

```
Step 1 ──→ Step 2 ──→ Step 3 (human) ──→ Step 4 (human)
                         │                     │
                    [승인/추가]           [승인/수정/재설계]
                         │                     │
                         ▼                     ▼
                    Step 2 (반복)         Step 5 ──→ Step 6 ──→ Step 7 (human)
                                                                    │
                                                              [승인/수정]
                                                                    │
                                                                    ▼
                                                              Step 8 (human)
                                                                    │
                                                                    ▼
                                                              Step 9 (Loop)
                                                               9a → 9b → 9c → 9d
                                                                    │       │
                                                               [FAIL→9b] [PASS]
                                                                         │
                                                                    Step 10 ──→ Step 11 (human) ──→ Step 12
                                                                    │                │
                                                               [치명적→9b]    [수정→9/5/4]
```

**역방향 루프의 설계 의도**: 품질이 기준에 미달하면 해당 단계뿐 아니라 **근본 원인이 있는 상위 단계로 복귀**할 수 있다. Step 11에서 구조적 문제가 발견되면 Step 5(스토리 바이블)나 Step 4(블루프린트)까지 거슬러 올라간다. 이것은 비용이 높지만, **절대 기준 1(품질)**이 명시적으로 속도보다 우선한다.

### 3.3 Human Gate의 전략적 배치

4개의 Human Gate가 파이프라인의 **전환점**에 배치된다:

| Gate | Step | 위치 의도 | 불가역성 수준 |
|------|------|---------|-------------|
| Step 3 | Research → Blueprint | 인터뷰 재료가 충분한지 판단 | 낮음 (추가 세션 가능) |
| Step 4 | Blueprint → Story Bible | **서사 구조의 최종 결정** | 높음 (이후 모든 작업이 이 구조에 종속) |
| Step 7 | Story Bible → Writing | **글쓰기 전 최종 검문** | 매우 높음 (챕터 전체 재작성 비용) |
| Step 11 | Writing → Build | **출판 전 최종 승인** | 최종 (되돌릴 수 없음) |

**Step 4가 가장 중요한 Gate인 이유**: 블루프린트(킥포인트, 핵심 메시지, 강조/배제 영역)는 이후 모든 단계를 **구속**한다. 잘못된 블루프린트 위에 지어진 스토리 바이블과 챕터는 아무리 잘 써도 구조적 결함이 있다. 이 단계에서 사용자의 적극적 참여가 필수적이다.

## 4. 에이전트 시스템

### 4.1 에이전트 설계 원리: P2 전문성 기반 위임

부모 게놈의 **P2 원칙**(전문성 기반 위임)이 자서전 도메인에서 강하게 발현된다:

```
Orchestrator (지휘자)
├── @interviewer      ← "듣는 전문가" (감각적 디테일 추출)
├── @story-architect  ← "지도 제작자" (삶의 지형을 구조화)
├── @chapter-writer   ← "투명한 목소리" (화자 뒤에 사라지는 작가)
├── @voice-calibrator ← "언어 계측기" (주관 아닌 수치로 측정)
├── @reviewer         ← "법의학적 비평가" (적대적 품질 감사)
├── @reviewer-deep    ← "동질화 탐지기" (반복 서사 제거)
└── @translator       ← "문화 번역가" (의미 보존 번역)
```

### 4.2 에이전트별 상세 아키텍처

#### @interviewer — "듣는 전문가"

```
정체성: "listener first, questioner second"
모델: opus | 도구: Read, Glob, Grep | 최대 턴: 40
```

**설계 근거**: 인터뷰어는 AI의 가장 인간적인 역할을 수행한다. 안전한 공간을 만들어 화자가 자연스럽게 기억을 풀어놓도록 이끈다. 기계적 질의응답이 아닌, "할머니 부엌의 냄새, 양철 지붕 위 빗소리, 결정적 순간에 누군가가 한 정확한 말"을 끌어낸다.

**핵심 메커니즘**:
- **최소 3세션 규칙**: 1세션=인생 시기 매핑, 2세션=심층 탐구, 3세션=교차 참조·갭 메우기
- **세션 후 상호작용 프로토콜** (강제): 매 세션 후 요약+커버리지 맵+3개 질문 → 사용자 확인 없이 절대 진행 불가
- **감정 마커 추출**: 인터뷰 중 감정적 반응을 `emotional_markers`로 태깅

#### @story-architect — "지도 제작자"

```
정체성: "cartographer of a life"
모델: opus | 도구: Read, Glob, Grep, Bash | 최대 턴: 30
SOT 쓰기: story_bible.json (유일한 비-Orchestrator 쓰기 권한)
```

**설계 근거**: 스토리 바이블은 모든 하류 에이전트가 의존하는 서사 SOT다. 한 명의 건축가가 전체 구조를 조망해야 테마 간 교차 연결이 약해지지 않는다. 복수의 건축가로 분할하면 "나무는 보지만 숲을 못 보는" 문제가 발생한다.

**핵심 메커니즘**:
- **블루프린트 구속**: 사용자가 Step 4에서 합의한 블루프린트(킥포인트, 핵심 메시지, 강조/배제)를 **반드시 준수**
- **보이스 가이드 기반**: `compute_voice_fingerprint.py`의 결정론적 메트릭을 baseline으로 사용, AI 추정치 사용 금지
- **소스 트레이싱**: 모든 사실에 원본 인터뷰 참조 태그

#### @chapter-writer — "투명한 목소리"

```
정체성: "the voice, not the author"
모델: opus | 도구: Read, Glob, Grep, Bash | 최대 턴: 30
```

**설계 근거**: 고스트라이터의 최고 덕목은 **투명성**이다. 독자가 고스트라이터의 존재를 잊어야 한다. 이를 위해 보이스 가이드의 수치 목표(문장 길이, 대화 비율, 수동태 비율 등)를 정밀하게 따른다.

**핵심 메커니즘**:
- **컨텍스트 조립**: `assemble_chapter_context.py`가 해당 챕터에 필요한 인터뷰·인물·이벤트만 필터링하여 제공
- **U자형 포지셔닝**: 보이스/스타일 지시를 프롬프트 시작에, 구체적 지시를 끝에 배치 (어텐션 분산 방지)
- **Show Don't Tell 원칙**: "그녀는 슬펐다" 대신 "그녀는 매년 생일마다 키를 표시했던 부엌 문틀 위의 홈을 손바닥으로 눌렀다"
- **스타일 블렌딩**: 선택된 작가 스타일과 화자의 고유 보이스를 확정된 비율로 혼합

#### @voice-calibrator — "언어 계측기"

```
정체성: "linguistic instrument, not a literary critic"
모델: sonnet | 도구: Read, Glob, Grep, Bash | 최대 턴: 25
```

**설계 근거**: sonnet 모델을 사용하는 유일한 에이전트다. 이유: 보이스 캘리브레이션은 **측정**이지 **판단**이 아니다. 문장 길이 분포, 대화 비율, 수동태 퍼센티지 등 정량적 메트릭만 계산한다. opus의 창의성이 필요 없고, sonnet의 정확한 실행력이 최적이다.

**분석 7차원**:
1. 문장 길이 분포 (평균, 표준편차, 히스토그램)
2. 어휘 수준 (type-token ratio)
3. 대화 분석 (대화:서술 비율, 대화 태그)
4. 수동태 탐지 (15% 임계값)
5. 부사 밀도 (1000단어당)
6. 금지어 체크 (보이스 가이드 금지 목록)
7. Show vs Tell 비율

#### @reviewer — "법의학적 비평가"

```
정체성: "forensic literary critic AND continuity editor"
모델: opus | 도구: Read, Write, Edit, Glob, Grep, Bash | 최대 턴: 40
```

**설계 근거**: 리뷰어는 도구 접근 권한이 가장 넓은 에이전트다 (Write, Edit 포함). 이유: 리뷰 결과를 파일로 직접 저장해야 하고, 필요 시 참조 파일을 수정하여 피드백을 기록한다.

**7차원 평가 프레임워크** (가중치 합산):

| 차원 | 가중치 | 평가 대상 |
|------|-------|---------|
| 감정적 공명 | 20% | 한-흥-정 전환, 체화된 감정, 여운 엔딩 |
| 보이스 일관성 | 20% | 화자다운 목소리, 경어 전환, 골든 예시 존재 |
| 산문 품질 | 15% | 번역체 부재, 침묵의 서술, 놀라운 문장 |
| 사실 정확성 | 15% | CoVe 검증, 출처 추적성, [INFERRED] 적절성 |
| 서사 아크 | 15% | 기승전결, 이중 의식 비율, 의미 발견 |
| 출처 추적성 | 10% | 인터뷰 참조 검증 가능, 날조 비율 상한 |
| 연속성 | 5% | CC-07 스레드 추적, 교차 챕터 일관성 |

**Chain-of-Verification (CoVe) 프로토콜**:
1. 챕터에서 모든 사실적 주장 추출
2. 각 주장을 Story Bible 대조 검증
3. Story Bible에서 확인 불가한 주장은 원본 인터뷰로 2차 검증
4. 검증 불가한 주장에 `[INFERRED]` 태그 부착

**범위 경계** (§22.5): 리뷰어는 **LLM만 할 수 있는 질적 판단**에 집중한다. 단어 수, 스키마, 금지어, 문장 길이 등 결정론적 검증은 **Python 스크립트가 사전에 완료**한다. 챕터가 리뷰어에게 도달했을 때, 기계적 검증은 이미 통과한 상태다.

#### @reviewer-deep — "동질화 탐지기"

```
정체성: anti-homogenization deep scan
모델: opus | 도구: Read, Glob, Grep | 최대 턴: 30
```

**설계 근거**: Step 10 전용. AI가 생성한 텍스트의 가장 은밀한 결함은 **동질화**(homogenization)다 — 모든 챕터가 비슷한 구조, 비슷한 전환, 비슷한 어미로 끝나는 현상. `@reviewer`가 개별 챕터의 품질을 평가하는 반면, `@reviewer-deep`은 **챕터 간 패턴 반복**을 탐지한다.

#### @translator — "문화 번역가"

```
정체성: English-to-Korean translation specialist
모델: opus | 도구: Read, Write, Glob, Grep | 최대 턴: 30
```

**설계 근거**: 번역은 단순한 언어 변환이 아니라 **문화 번역**이다. 7단계 번역 프로토콜(1차 번역 → 자기 검토 → 용어 사전 대조 → 최종본)과 `translations/glossary.yaml` 용어 사전으로 일관성을 보장한다.

### 4.3 에이전트 간 데이터 계약

에이전트 간 통신은 **파일 기반** 계약으로 이루어진다. 직접 메시지 교환이 아닌, JSON Schema로 검증된 산출물 파일을 통해 소통한다:

```
@interviewer ──[INT-{NNN}.json]──→ @story-architect
                                        │
                              [story_bible.json]
                                        │
                           ┌────────────┼────────────┐
                           ▼            ▼            ▼
                  @chapter-writer  @reviewer   @reviewer-deep
                        │
               [ch{NN}_draft.md]
                        │
                 @voice-calibrator
                        │
               [calibration report]
                        │
                   @reviewer
                        │
               [RV-ch{NN}.json]
```

**계약 집행 메커니즘**: `schema_validator.py`가 모든 JSON 산출물의 스키마 적합성을 강제한다. 스키마 위반 시 파이프라인이 멈춘다.

### 4.4 기술 선택: Sub-agent vs Agent Team

| 단계 | 기술 | 근거 |
|------|------|------|
| Step 2 (인터뷰) | Sub-agent (순차) | 라포 형성에 대화 연속성 필수 |
| Step 4 (블루프린트) | Sub-agent (순차) | 사용자 피드백 반영에 단일 대화 스레드 필요 |
| Step 5 (스토리 바이블) | Sub-agent | 전체 조망에 단일 아키텍트 비전 필요 |
| Step 8 (스타일 선택) | Sub-agent (순차) | 캘리브레이션 샘플이 순차적으로 개선되어야 함 |
| Step 9 (챕터 작성) | **조건부**: ≤6장=Sub-agent, >6장=Agent Team | 500KB 이상 컨텍스트에서 어텐션 피로 발생 |
| Step 10 (교차 일관성) | Agent Team (차원별) | 독립적 리뷰어가 다른 결함을 잡음 |

## 6. 보이스 & 스타일 시스템

### 6.1 설계 철학: "느낌"이 아닌 "수치"

자서전에서 가장 중요한 것은 **화자의 목소리**다. 이 시스템의 보이스 시스템은 하나의 원칙에 기반한다:

> **보이스 일관성은 주관적 감상이 아닌 정량적 메트릭으로 측정하고 강제한다.**

이 원칙이 필요한 이유:
- LLM은 프롬프트만으로 "이 사람처럼 써라"고 지시받으면 **세션마다 다른 해석**을 한다
- 12개 챕터에 걸쳐 일관된 보이스를 유지하려면 **수치 기준선**이 필요하다
- "느낌이 다르다"는 피드백은 작가에게 구체적 방향을 주지 못한다

### 6.2 보이스 파이프라인

```
Step 2-3: 인터뷰 수집
    │
    ▼
compute_voice_fingerprint.py  ◄── 결정론적 Python 계산
    │
    ├── avg_sentence_length: 14.2 words
    ├── dialogue_ratio: 0.35
    ├── passive_voice_pct: 0.08
    ├── type_token_ratio: 0.72
    ├── adverb_density: 12.3 per 1000
    ├── show_tell_ratio: 0.65
    └── forbidden_words: ["very", "really", ...]
    │
    ▼
Step 5: @story-architect → story_bible.voice_guide
    │   (Python 메트릭을 baseline으로 사용, AI 추정 금지)
    │
    ▼
Step 8: 사용자가 10종 스타일 중 선택
    │   + 블렌딩 비율 결정 (기본: 30% 스타일 / 70% 보이스)
    │
    ▼
style_blender.py --apply  ◄── 순수 산술
    │
    │   blended_param = skill_param × ratio + observed_param × (1 - ratio)
    │
    ├── avg_sentence_length: 14.2 → 16.8 (Hemingway 영향)
    ├── dialogue_ratio: 0.35 → 0.32
    └── ...
    │
    ▼
Step 9a: assemble_chapter_context.py
    │   U자형 포지셔닝: 보이스/스타일을 START에, 지시를 END에
    │
    ▼
Step 9b: @chapter-writer가 블렌딩된 수치 목표를 따라 산문 작성
    │
    ▼
Step 9c: @voice-calibrator가 산문의 실제 메트릭을 계산하여 편차 보고
    │
    ▼
Step 9d: validate_voice_match.py (VM1-VM6)  ◄── 결정론적 최종 검증
```

### 6.3 스타일 블렌딩의 수학

`style_blender.py`의 블렌딩은 **단순 가중 평균**이다:

```python
blended_value = style_value * blending_ratio + voice_value * (1 - blending_ratio)
```

**예시** (블렌딩 비율 = 0.3):
- 화자의 평균 문장 길이: 14.2 단어
- Hemingway 스타일 목표: 8.0 단어
- 블렌딩 결과: 8.0 × 0.3 + 14.2 × 0.7 = **12.34 단어**

이 산술이 **AI 추정치가 아닌 결정론적 계산**이라는 점이 핵심이다. `_blend_metadata.computed_by = "style_blender.py"`로 출처를 명시한다.

### 6.4 10종 스타일 갤러리

각 스타일은 `.claude/skills/writing-styles/style-{name}/SKILL.md`에 정의되며, 다음을 포함한다:
- 핵심 기법과 문학적 철학
- 보이스 파라미터 수치 (문장 길이, 대화 비율 등)
- 적합한 자서전 유형
- 골든 예시 문장 2개

| 스타일 | 핵심 기법 | 대표 메트릭 |
|--------|---------|-----------|

## 6a. 한국 문학 품질 시스템 (Appendix A Literary Directives)

### 6a.1 설계 철학: 자서전은 기록이 아니라 문학이다

> **이 시스템이 생산하는 것은 '인생 기록'이 아니라 '읽을 가치가 있는 문학 작품'이다.**

이 원칙을 강제하기 위해 11개의 문학 지시문(Appendix A)이 `@chapter-writer`와 `@reviewer`에 비협상적(non-negotiable) 의무로 내장된다. 지시문은 한국 문학 전통의 미학을 코드화한 것이다.

### 6a.2 감정 아키텍처: 한-흥-정 (恨-興-情)

한국 서사 전통의 감정 구조를 자서전에 구조적으로 적용한다:

| 감정 모드 | 의미 | 서사 표현 방법 | **표현 금지** |
|----------|------|-------------|-------------|
| **한(恨)** | 축적된 슬픔, 미해결 원한 | 물리적 무게, 침묵, 반복, 너무 오래 간직한 사물 | "어머니는 한이 서려 있었다" |
| **흥(興)** | 격앙된 기쁨 | 리듬, 공동체 행위, 자발적 움직임 | "모두가 흥에 겨워 춤을 추었다" |
| **정(情)** | 깊은 대인적 유대 | 습관적 보살핌, 말없는 이해, 작은 행위 | "할머니와 나 사이에는 정이 깊었다" |

**핵심 규칙**: 한, 흥, 정이라는 **단어 자체는 산문에서 감정 라벨로 절대 사용 금지**. 체화(embodied)된 장면으로만 전달한다. `@story-architect`가 인물의 인생을 한-흥-정 사이클로 매핑하고, 전환점을 서사 전환점으로 활용한다.

### 6a.3 서사 구조: 기승전결 (起承轉結)

4막 리듬이 **챕터 단위**와 **전체 원고 단위** 이중으로 적용된다:

| 막 | 비율 | 기능 | **핵심 구분** |
|----|------|------|-------------|
| 기(起) | ~20% | 장면 설정, 독자 안착 | |
| 승(承) | ~30% | 복잡성 발전, 긴장 | |
| 전(轉) | ~30% | **관점 전환** | 클라이맥스가 아님 — 깨달음, 아이러니, 재맥락화 |
| 결(結) | ~20% | 새로운 이해에 안착, 여운 엔딩 | |

### 6a.4 이중 의식 (Experiencing vs Narrating Self)

자서전의 독특한 서사 장치 — "과거의 나"와 "지금의 나"의 비율이 인생 시기에 따라 달라진다:

| 인생 시기 | 체험하는 자아 : 서술하는 자아 |
|----------|--------------------------|
| 어린 시절 (0-11) | 80 : 20 |
| 청소년 (12-17) | 70 : 30 |
| 청년 (18-30) | 60 : 40 |
| 중년 (31-55) | 50 : 50 |
| 노년 (56+) | 30 : 70 |

### 6a.5 장면 구성 6단계 (챕터당 최소 2장면)

```
핫스팟 → 장소 → 감각 → 대화 → 신체화 → 의미
```

1. **핫스팟**: 인터뷰에서 depth_score > 0.7인 장면에서 시작
2. **장소**: 특정 위치에 감각 디테일 ≥2개로 정착
3. **감각**: 챕터의 `texture` 감각 양식 우선 적용
4. **대화**: 인터뷰 실제 인용 사용, 없으면 시대 적합한 대화 구성
5. **신체화**: 감정을 신체 감각/제스처로 표현, 추상적 서술 금지
6. **의미**: 장면 자체에서 의미가 출현하도록 — 해석 첨부 금지

### 6a.6 번역체 9패턴 금지 (A-10)

AI가 생성한 한국어 텍스트의 가장 흔한 결함은 **번역체**(translationese)다. 9개 패턴을 원천 차단한다:

| # | 금지 패턴 | 예시 (BAD → GOOD) |
|---|---------|-----------------|
| 1 | 피동 되다 연쇄 | "결정이 내려지게 되었다" → "결정을 내렸다" |
| 2 | 것이다 문말 | "중요한 것이다" → "중요했다" |
| 3 | ~에 있어서 | "교육에 있어서" → "교육에서" |
| 4 | ~(으)로서의 | "아버지로서의 역할" → "아버지 노릇" |
| 5 | ~적(的) 남발 | "역사적 사회적 경제적" → 각각 풀어서 |
| 6 | 불필요한 그것/이것 | "그것은 중요한 문제였다" → "중요한 문제였다" |
| 7 | 영어식 수동태 | "그는 존경을 받았다" → "사람들이 그를 존경했다" |
| 8 | ~하는 것 명사화 | "공부하는 것을 좋아했다" → "공부를 좋아했다" |
| 9 | 과도한 한자어 | "인지(認知)하다" → "알아차리다" |

패턴 1-4는 `@reviewer`가 질적으로, 패턴 5-9는 Python regex가 결정론적으로 검증한다.

### 6a.7 Golden Exemplar 시스템

`@story-architect`가 인터뷰에서 **비범하게 생동감 있는 3-5개 구절**을 추출하여 Golden Exemplar로 지정한다:

- **선정 기준**: 감각 디테일 밀도, 메타포, 감정 강도, 길이 (depth_score > 0.7)
- **용도**: `@chapter-writer`가 핵심 장면 작성 전 **톤 앵커**로 참조
- **검증**: `@reviewer-deep`의 DC-6 체크 — 챕터당 ≥1 Golden Exemplar (50-200단어, 기저선보다 현저히 생동감 있는 구절)

### 6a.8 허용 윤색 프로토콜 (A-11)

| 허용 | 금지 |
|------|------|
| 대화 재구성 (요지 기억, 정확한 말 아님) | 일어나지 않은 사건 날조 |
| 묘사된 배경과 일치하는 감각 디테일 추가 | 표현/암시되지 않은 감정 귀속 |
| 역사적으로 정확한 환경의 배경 디테일 채우기 | 언급되지 않은 인물 창조 |
| | 결과 변경, 미검증 역사 주장 |

모든 윤색에 `[INFERRED]` 태그: type, description, justification, source_interview.

### 6b.1 4중 방법론 융합

`@interviewer`는 4가지 학술 프레임워크를 통합한 구조화된 인터뷰를 수행한다:

| **Rubin Sensory Cues** | 비자발적 기억 활성화 | 세션당 ≥1 감각 단서 질문 |

| `emotional_markers[]` | 감정 강도가 변하는 순간 (눈물, 웃음, 목소리 변화, 멈춤, 회피, 활기) |

| `places_mentioned[]` | 이름, 유형, 시기, 감각 디테일 |

### 6b.3 세션 마무리 의식 (한국어, 강제)

| 사실 날조 | 소스 트레이싱 — 모든 사실에 인터뷰 참조 태그 | `validate_source_tracing.py` |
| 보이스 일탈 | Python 계산 메트릭 기반 보이스 가이드 | `compute_voice_fingerprint.py` + `validate_voice_match.py` |
| 스토리 바이블 모순 | 엔티티·타임라인·캐릭터 교차 검증 | `validate_story_bible.py` |
| 교차 챕터 모순 | 전체 원고 8차원 일관성 감사 | `validate_consistency.py` |
| pACS 자기 과대평가 | 바닥 점수 계산 + 이전 점수 추이 분석 | `validate_pacs_floor.py` |
| 블루프린트 무시 | 스토리 바이블이 블루프린트 구속을 준수하는지 검증 | `validate_blueprint.py` |
| 추론 표시 누락 | `[INFERRED]` 태그 비율 검증 | `validate_source_tracing.py` (ST 체크) |

## 8. 프로젝트 격리

### 8.1 격리 필요성

여러 자서전 프로젝트를 동시에 진행하거나 순차적으로 진행할 때, 산출물이 뒤섞이면:
- A 인물의 인터뷰가 B 인물의 챕터에 영향
- 이전 프로젝트의 스토리 바이블이 현재 프로젝트를 오염
- SOT가 다른 프로젝트의 상태를 반영

### 8.2 격리 메커니즘

```
autobiography-generator/
├── projects/
│   ├── 20260318-최윤식/          ← 프로젝트 A
│   │   ├── outputs/              (산출물)
│   │   └── review-logs/          (리뷰 로그)
│   └── 20260320-김영희/          ← 프로젝트 B
│       ├── outputs/
│       └── review-logs/
├── outputs → projects/20260318-최윤식/outputs     ← symlink (활성 프로젝트)
└── review-logs → projects/20260318-최윤식/review-logs  ← symlink
```

**핵심 설계**:
- `init_project.py`가 프로젝트 ID(`YYYYMMDD-이름`), 디렉터리, symlink을 **원자적으로** 생성
- 기존 스크립트 13종은 변경 없음 — symlink이 `os.path.abspath()` 수준에서 **투과**
- SOT의 `project.id`와 `project.dir`가 활성 프로젝트를 식별
- `validate_project_init.py`가 PI1-PI8 검증 (디렉터리, symlink, SOT 정합성)

## 9. 번역 파이프라인

### 9.1 English-First + Korean Pair 원칙

```
모든 에이전트는 영어로 작업  ──→  @translator가 한국어 번역을 생산
```

**영어 우선의 근거**: LLM의 영어 성능이 한국어보다 현저히 높다 (절대 기준 1 근거). 복잡한 서사 구조, 감정적 뉘앙스, 적대적 리뷰 등 고급 언어 작업에서 영어가 한국어보다 정확하다.

### 9.2 번역 프로토콜

`@translator`의 7단계 프로토콜:

1. 원문 분석 — 구조, 톤, 핵심 용어 식별
2. 1차 번역 — 의미 보존 중심
3. 자기 검토 — 번역체 패턴 제거
4. 용어 사전 대조 — `translations/glossary.yaml` 일관성
5. 문화적 맥락화 — 한국어 독자에 맞는 표현
6. 최종 교정
7. 품질 자기 평가

### 9.3 번역 검증

`validate_translation.py`가 T1-T9 + RV1 체크로 번역 품질을 결정론적으로 검증한다. 번역체 패턴 5-9(수동태, SVO 순서, 접속사 과용 등)는 regex로 자동 탐지된다.

## 10. 기술 선택과 근거

### 10.1 Python 검증 스크립트 (30종+)

**왜 Python인가**: LLM에게 "이 JSON이 올바른지 확인해"라고 요청하면 할루시네이션이 발생한다. Python 스크립트는 **결정론적**이다 — 같은 입력에 항상 같은 결과. 이것이 P1(할루시네이션 봉쇄) 원칙의 기술적 구현이다.

### 10.2 JSON Schema (6종)

**왜 Schema인가**: 에이전트 간 데이터 계약을 **코드로 강제**한다. "이 필드가 있어야 한다"는 문서가 아니라 런타임 검증이다. `schema_validator.py`가 모든 JSON 쓰기 후 자동 실행된다.

### 10.3 YAML SOT

**왜 YAML인가**: `state.yaml`은 사람이 읽고 디버깅할 수 있어야 한다. JSON보다 가독성이 높고, 주석을 달 수 있다. PyYAML로 Python에서 자연스럽게 파싱된다.

### 10.4 Pandoc + XeLaTeX

**왜 이 스택인가**: Step 12의 책 빌드에 사용. Pandoc은 Markdown → LaTeX → PDF 변환의 표준 도구이고, XeLaTeX은 유니코드(한국어) 지원이 우수하다. memoir 클래스는 책 조판에 최적화되어 있다.

### 10.5 Hook 시스템

부모 게놈에서 유전된 14개 Hook이 자서전 도메인에서도 작동한다:

| 계층 | Hook 수 | 대표 예시 |
|------|---------|---------|
| 안전 (Safety) | 5 | 위험 명령 차단, TDD Guard, 시크릿 탐지 |
| 검증 (Validation) | 9 | pACS, 리뷰, 추적성, 도메인 지식 |
| 관측성 (Observability) | 2 | 대시보드, 진단 |
| 컨텍스트 보존 | 6 | 저장, 복원, 증분 스냅샷 |

자서전 전용 Hook 3종이 추가로 사용 가능(미활성):
- `update_state_on_write.py` — 산출물 쓰기 시 자동 SOT 갱신
- `track_chapter_progress.py` — 챕터 진행 메트릭 로깅
- `checkpoint_state.py` — 상태 체크포인트 (context_guard.py에 의해 대체됨)

### 11.1 부모 게놈 (AgenticWorkflow)에서 유전된 요소

### 11.2 도메인 특화 유전자 발현

부모 게놈에는 있지만 자서전 도메인에서 **특히 강하게 발현**되는 유전자:

1. **P2 전문성 위임**: 8종 에이전트가 각각 고유한 전문성(인터뷰, 서사 구조, 산문 작성, 보이스 분석, 비평)을 담당
2. **Generator-Critic**: 챕터 수준과 원고 수준에서 이중으로 작동
3. **Human Gate**: 4곳에서 사용자 개입을 보장하여 AI의 한계를 인간이 보완

### 11.3 자서전 도메인 고유 유전자

부모 게놈에 없고, 자서전 도메인에서 **새로 발현**된 유전자:

| 유전자 | 역할 | 존재 이유 |

## 12. 주요 설계 결정

### ADR-1: 영어 우선 워크플로우

- **결정**: 모든 에이전트가 영어로 작업, `@translator`가 한국어 번역 생산
- **근거**: LLM의 영어 성능이 한국어보다 현저히 우수 (절대 기준 1)
- **대안 검토**: 한국어 직접 생성 → 산문 품질 저하로 기각
- **트레이드오프**: 번역 비용(토큰·시간) 증가 → 품질 절대주의가 비용을 정당화

### ADR-2: 보이스 파라미터의 결정론적 계산

- **결정**: `compute_voice_fingerprint.py`가 인터뷰에서 보이스 메트릭을 Python으로 계산
- **근거**: AI에게 "이 사람의 보이스를 분석해"라고 하면 세션마다 다른 결과 (P1)
- **대안 검토**: LLM 기반 보이스 분석 → 재현 불가로 기각
- **트레이드오프**: 정량화할 수 없는 뉘앙스 손실 → `@reviewer`의 질적 평가로 보완

### ADR-3: 스토리 블루프린트의 구속력

- **결정**: Step 4에서 사용자와 합의한 블루프린트가 이후 모든 단계를 **구속**
- **근거**: 자서전은 화자의 이야기이며, 서사 구조는 화자가 결정해야 한다
- **대안 검토**: AI가 최적 구조를 자동 결정 → 화자의 주권 침해로 기각
- **트레이드오프**: AI가 더 좋은 구조를 알더라도 사용자 결정을 따름 → 인간 주체성 보존

### ADR-4: @voice-calibrator에 sonnet 모델 사용

- **결정**: 8종 에이전트 중 유일하게 sonnet 모델 사용
- **근거**: 보이스 캘리브레이션은 측정(수치 계산)이지 판단(창의적 평가)이 아님
- **대안 검토**: opus 모델 → 불필요한 비용 증가, 정확성 차이 미미
- **트레이드오프**: 측정의 정밀도 ↔ 비용 효율성 → 균형점

### ADR-5: 챕터 6장 기준 Sub-agent/Agent Team 전환

- **결정**: 총 챕터 ≤ 6이면 Sub-agent(순차), > 6이면 Agent Team(병렬)
- **근거**: 500KB 이상 컨텍스트에서 LLM의 어텐션 피로(attention fatigue)가 보이스 일관성 저하를 유발
- **대안 검토**: 항상 Agent Team → 6장 이하에서는 오히려 보이스 불일치 증가
- **트레이드오프**: 병렬화 ↔ 보이스 일관성 → 컨텍스트 크기 기준으로 전환

### ADR-6: 리뷰어의 anti-rubber-stamp 규칙

- **결정**: `@reviewer`는 최소 1개 이슈를 반드시 발견해야 함
- **근거**: "문제 없음" 리뷰는 의미 없는 고무 도장(rubber stamp)이다
- **대안 검토**: 이슈 없으면 PASS → 리뷰의 가치 하락으로 기각
- **트레이드오프**: 사소한 이슈까지 보고 → 리뷰의 실질적 가치 보존
