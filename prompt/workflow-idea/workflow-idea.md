## 제1 핵심 목적 (절대 목표)

> **코딩 경험 제로인 사역자가 Claude Code와 한국어 대화만으로, 중학생이 "이거 진짜 앱이다!"라고 느끼는 수준의 수련회 앱을 처음부터 끝까지 자동으로 완성한다.**

이 목적은 workflow의 모든 단계와 모든 기능에 '절대 목표'로 고정된다.

검증 질문 (모든 설계 결정에 적용):
> 1. "코딩을 모르는 사역자가 이 단계에서 막히지 않는가?"
> 2. "완성된 결과물을 중학생이 보고 '이거 진짜 앱이다'라고 느끼는가?"

## workflow.md의 정체성 (성찰 E2 신설)

> **이 섹션은 workflow.md가 "무엇"인지를 정의한다. 이것이 명확하지 않으면 workflow.md 작성 자체가 불가능하다.**

### workflow.md란?

```
workflow.md는 Claude Code가 읽고 따르는 "지시서"이다.

  - SaaS 제품의 설계도가 아니다.
  - 별도의 소프트웨어 시스템을 구축하는 것이 아니다.
  - Claude Code라는 기존 도구가 "이 문서를 읽고"
    사역자와 대화하면서 앱을 자동으로 만들어주는 것이다.

workflow.md의 역할:
  사역자가 Claude Code를 실행한다
  → Claude Code가 workflow.md를 읽는다
  → workflow.md에 적힌 대로 사역자와 대화한다
  → workflow.md에 적힌 대로 코드를 생성하고 서버를 실행한다
  → 사역자는 workflow.md의 존재를 모른다 (내부 지시서)
```

### AgenticWorkflow 프로젝트 구조와의 매핑

```
이 프로젝트(AgenticWorkflow)에는 이미 다음 구조가 있다:

  CLAUDE.md          — Claude Code 최상위 지시서
  .claude/agents/    — 서브에이전트 정의 파일
  .claude/hooks/     — 자동 실행 스크립트
  .claude/skills/    — 스킬 정의 (workflow-generator 등)

workflow.md는 이 구조 안에서 다음과 같이 위치한다:

  [방식 A] 독립 workflow.md 파일
    → Claude Code가 세션 시작 시 이 파일을 읽고 지침을 따른다
    → CLAUDE.md에서 workflow.md를 참조하도록 설정

  [방식 B] .claude/skills/ 내 스킬로 통합
    → workflow-generator 스킬이 workflow.md를 생성
    → 생성된 workflow.md를 Claude Code가 실행

  → 최종 방식은 workflow.md 작성 단계에서 확정
```

### 절대 제약: 로컬 컴퓨터 실행

```
workflow.md가 지시하는 모든 작업은 사용자의 로컬 컴퓨터에서 실행된다.

  ✅ 허용: 사역자 PC에서 파일 생성, 코드 작성, 서버 실행
  ✅ 허용: 같은 WiFi/핫스팟 내 LAN 접속
  ✅ 허용: Claude API 호출 (Claude Code 자체 동작에 필요)
  
  ❌ 금지: 외부 서버/서비스를 경유하는 데이터 전송
  ❌ 금지: 제3자 SaaS 서비스에 의존하는 기능
  ❌ 금지: 사역자 동의 없이 인터넷에 데이터 노출

  GitHub Pages는 사역자가 명시적으로 동의한 경우에만 사용 (선택 사항).
  기본 동작은 항상 로컬 컴퓨터 + LAN이다.
```

## 1. 워크플로우 구조: 7단계 Phase (Phase 0~6)

### 아키텍처 개요 (성찰 E3 전면 개정)

> **핵심 변경**: "에이전트 팀"이라는 용어가 혼란을 유발했다.
> Claude Code는 **단일 대화 세션**에서 실행된다. 별도의 에이전트 프로세스가 아니다.
> 따라서 "에이전트 팀"이 아니라 **"역할 단계"**로 재정의한다.

```
실행 주체: Claude Code (단일 대화 세션)

사역자 (한국어 대화)
       |
Claude Code가 workflow.md를 읽고,
아래 4가지 역할을 순서대로 수행한다:
       |
[역할 1: 대화 진행] --- 사역자 의도 파악 + 콘텐츠 수집
       |
[역할 2: 코드 생성] --- HTML/CSS/JS 작성, 서버 코드
       |
[역할 3: 품질 검증] --- Q1~Q10, D1~D6 자동 체크
       |
[역할 4: 배��� 실행] --- LAN 서버 실행, QR 생성
```

**"에이전트"가 아닌 이유**:
- Claude Code의 Agent tool로 서브에이전트를 spawn할 수는 있으나,
  이 워크플로우에서 서브에이전트가 반드시 필요한 것은 아니다.
- 서브에이전트 사용 여부는 **workflow.md 작성 시** 성능/안정성을 고려하여 결정한다.
- 아이디어 단계에서는 **"역할 분리"**만 확정하고, 실행 메커니즘은 열어둔다.

**역할 간 전환 = 파일 기반 상태 저장**:
```
각 역할 단계 완료 시 산출물을 프로젝트 폴더에 파일로 저장한다.

  [대화 진행] 완료 -> 프로젝트 폴더 내 상태 파일에 의도/콘텐츠 기록
  [코드 생성] 완료 -> 프로젝트 폴더 내 실제 코드 파일 + Git 커밋
  [품질 검증] 완료 -> 프로젝트 폴더 내 검증 결과 기록
  [배포 실행] 완료 -> 서버 URL, QR 경로 기록

  Claude Code 세션이 끊어져도 파일은 남아 있으므로 재개 가능.
  구체적인 파일 형식(JSON, YAML 등)은 workflow.md 작성 시 확정.
```

**컨텍스트 윈도우 초과 대응** (성찰 G9 신설):
```
문제: 퀴즈 50문제 + 디자인 반복 + 코드 생성이 쌓이면
      Claude Code 컨텍스트 윈도우가 초과될 수 있다.
      초과 시 이전 결정사항, 파일 구조 등을 망각한다.

대응 전략:
  [1] 핵심 결정만 상태 파일에 기록
      -> 앱 유형, 팀 구성, 색상 팔레트, 콘텐츠 요약 등
      -> 컨텍스트 압축 시에도 파일을 다시 읽으면 복원 가능

  [2] 대량 콘텐츠는 대화가 아닌 파일로 입력
      -> 퀴즈 50문제를 대화로 받으면 컨텍스트를 잡아먹음
      -> "메모장에 적어서 저장해주세요" 안내 -> 파일로 읽기
      -> 대화 컨텍스트에는 "퀴즈 50문제 파일에서 로드 완료"만 남김

  [3] Claude Code의 컨텍스트 보존 시스템 활용
      -> AgenticWorkflow 프로젝트에 이미 context preservation 시스템 존재
      -> 세션 압축/중단 시 자동 스냅샷 저장 -> 재개 시 복원
```

### Phase 상세

| Phase | 이름 | 수행 주체 | 사역자 역할 |
|-------|------|-----------|------------|
| 0 | 환경 사전 준비 | **기술 도우미 + Claude 안내** | 없음 (1회, 도우미가 수행) |
| **1** | **대화 & 콘텐츠** | **Claude Code (역할 1: 대화 진행)** | **원하는 앱 설명 + 콘텐츠 입력** |
| 2 | 프로젝트 초기화 | Claude Code (역할 2: 코드 생성) | 없음 (기다리기) |
| 3 | 코드 생성 | Claude Code (역할 2: 코드 생성) | 없음 (진행률만 확인) |
| 4 | 품질 검증 | Claude Code (역할 3: 품질 검증) | 없음 (자동) |
| 5 | 미리보기 & 수정 | Claude Code **(역할 1+2 공동)** + 사역자 | **사역자가 참여** |
| 6 | 배포 | Claude Code (역할 4: 배포 실행) | QR코드 받기 |

> **성찰 E4 반영**: 기존 Phase 1(의도 파악)과 Phase 2(콘텐츠 수집)를 **Phase 1(대화 & 콘텐츠)**로 통합했다.
> 사역자 관점에서 "어떤 앱을 만들지"와 "퀴즈 문제 뭐 넣을지"는 하나의 자연스러운 대화 흐름이다.
> 이것을 별도 Phase로 분리하면 불필요한 확인 절차가 생겨 "대화만으로" 원칙에 역행한다.
> 내부적으로는 의도 파악 -> 콘텐츠 수집 순서를 유지하되, 사역자에게는 끊김 없는 하나의 대화로 보인다.
>
> **성찰 E3 반영**: "에이전트 팀" 용어를 "역할"로 변경. Claude Code가 단일 세션에서 역할을 전환하며 수행.

```

### 핵심 포인트

사역자가 실질적으로 개입하는 구간은 **Phase 1(대화 & 콘텐츠)과 Phase 5(피드백)** 2곳. 나머지는 완전 자동.

## 2. 역할 단계 정의 (성찰 E3 전면 개정)

> **Claude Code가 단일 대화 세션에서 4가지 역할을 순서대로 수행한다.**
> "에이전트 팀"이 아니라 **"역할 전환"**이다. 실행 주체는 항상 Claude Code 하나.

### 역할 1: 대화 진행 (Phase 0~1)

- 사역자 대화 관리 (의도 파악, 확인 신호 판단)
- 앱 유형 감지 + 필요 정보 재질문 + 콘텐츠 수집 (Phase 1 통합)
- 에러 발생 시 사역자에게 보이는 메시지 관리 (한국어, 기술 용어 금지)
- 각 Phase 완료 시 명시적 단계 전환 출력: "준비가 끝났어요, 만들기 시작할게요"
- 이전 앱 재사용 요청 감지 (성찰 C3)

### 역할 2: 코드 생성 (Phase 2~3)

- HTML 구조 + 한국어 콘텐츠 삽입
- CSS (모바일 우선 375px, 44px 터치 영역, 다크모드, 디자인 시스템)
- JavaScript 로직 (XSS 방어, 오프라인 캐시, 실시간 기능)
- 9개 앱 유형별 코드 직접 작성 (템플릿이 아닌 대화 기반 생성)
- 수정 요청 시 최소 범위 변경 (색상 변경에 전체 재생성 안 함)
- 300KB 번들 제한 준수
- **모든 앱에 결과 저장 기능 기본 포함** (성찰 C2)
- **"앱 실행.bat" 파일 자동 생성** (성찰 C1, E5, E6)

### 역할 3: 품질 검증 (Phase 4)

**기술 품질 게이트 (Q1~Q10)**:
- Q1: 서버 실행 (HTTP 200 응답)
- Q2: HTML 유효성
- Q3: 외부 의존성 (CDN 폰트 외 0개)
- Q4: 번들 크기 (300KB 이하)
- Q5: 한국어 렌더링
- Q6: 터치 영역 (44x44px)
- Q7: QR 코드 URL 정확성
- Q8: 관리자 보호 (/admin 비밀번호)
- Q9: XSS 방지
- Q10: 사역자 시각적 확인 (유일한 수동 검증)

**디자인 품질 게이트 (D1~D6)** — 성찰 A1:
> → **상세 기준은 SS8(디자인 품질 보장 전략) 참조** (성찰 G4: SOT 단일화)
> D1: 카드 UI / D2: 애니메이션 / D3: 다크모드 / D4: 색상 일관성 / D5: 모바일 네이티브 / D6: 폰트 가독성

**검증 흐름**:
- Q1~Q9 + D1~D6: Claude Code가 자동 실행. 모두 통과해야 사역자에게 미리보기 제공.
- Q10: 사역자 확인. 사역자가 "좋아요" -> 배포 단계 진행.
- 실패 시: 자동 수정 후 재검증 (최대 3회). 3회 실패 시 롤백 후 사역자에게 보고.

### 역할 4: 배포 실행 (Phase 6)

- **LAN 서버 백그라운드 실행** (성찰 E5 — 아래 상세)
- WiFi IP 자동 감지 (일반 WiFi / 모바일 핫스팟)
- QR 코드 PNG 생성 + 인쇄용 HTML (A4)
- 브라우저 자동 열기
- GitHub Pages 배포 (선택 사항, 사역자 명시적 동의 시에만)
- 환경 감지: 포트(3000~3009 + 8080, 49152~49162 고포트), 디스크 공간, 한글 경로, 방화벽
- **"앱 실행.bat" 바탕화면 자동 배치** (성찰 C1, E6)

> **성찰 E1 반영**: 터널링(ngrok 등) 옵션을 **전면 삭제**했다.
> ngrok은 외부 서버를 경유하므로 "로컬 컴퓨터에서 작동" 절대 원칙에 위배된다.
> 네트워크 문제는 로컬 범위 내에서만 해결한다 (WiFi + 핫스팟).

## 3. 대화 설계: "길을 잃지 않는 대화"

### 대화 & 콘텐츠 (Phase 1 — 성찰 E4 통합)

> Phase 1은 의도 파악과 콘텐츠 수집이 **하나의 자연스러운 대화 흐름**으로 진행된다.
> 사역자는 Phase가 나뉘어 있다는 것을 모른다.

**"수련회 앱 메뉴판"으로 시작** (성찰 G6 신설):
> 사역자는 "앱으로 뭘 할 수 있는지" 모른다.
> 빈 질문("뭘 만들까요?")이 아니라, 구체적 예시를 먼저 보여준다.

```
Claude Code 첫 대화:
  "안녕하세요! 수련회 앱을 만들어드릴게요.
   이런 앱들을 만들 수 있어요:

   1) 성경 퀴즈 대회 — 팀끼리 버저 누르며 경쟁
   2) 팀 점수판 — 빔프로젝터에 실시간 순위 표시
   3) 수련회 일정표 — 학생 폰에서 일정 확인
   4) 찬양 가사 — 빔이랑 폰에 동시에 가사 표시
   5) 미션 스탬프 랠리 — QR코드로 미션 인증
   6) 전부 합친 종합 앱

   어떤 게 필요하세요? 여러 개 골라도 돼요!"
```

**대화 흐름**:
```
사역자 발화 -> Claude가 분류
  |-- 메뉴에서 선택 -> 바로 필요한 콘텐츠 질문으로 이어감
  |-- 명확한 요청 -> 바로 필요한 콘텐츠 질문으로 이어감
  |-- 모호한 요청 -> 메뉴판 다시 제시 + 구체적 선택지 2~3개
  |-- 범위 밖 요청 -> 이유 설명 + 대안 제시 (거절하지 않음)
  |-- "작년 앱 다시 쓰고 싶다" -> 이전 앱 불러오기 (성찰 C3)
```

**대화 예시 (의도 파악 -> 콘텐츠 수집이 자연스럽게 이어지는 흐름)**:
```
사역자: "성경 퀴즈 앱 만들어주세요"
Claude: "좋아요! 몇 가지만 알려주세요.
        팀이 몇 개인가요?"
사역자: "4개요"
Claude: "퀴즈 문제는 미리 준비하셨나요,
        아니면 제가 성경 범위를 알려주시면 만들어드릴까요?"
사역자: "요한복음에서 10문제 만들어주세요"
Claude: "알겠어요! 이렇게 만들게요.
        [구조 미리보기]
        시작할까요?"
```

**규칙**:
- 질문은 한 번에 최대 2개, 항상 구체적 보기 포함
- 기술 용어 완전 배제 ("JSON", "npm", "포트", "빌드" 금지)
- 진행률 표시: "[1/3단계] 어떤 앱인지 파악 중"

**앱 유형별 필수 콘텐츠** (대화 중 자연스럽게 수집):

| 앱 유형 | 필수 콘텐츠 | 수집 방식 |
|---------|------------|-----------|
| 성경 퀴즈 | 퀴즈 문제 + 정답 + 팀 구성 | 소량: 대화 / 대량: 메모장 파일 |
| 스탬프 랠리 | 미션 장소 + 미션 내용 | 대화 |
| 일정표 | 날짜별 일정 + 장소 | 대화 (자연어) |
| 찬양 가사 | 곡명 + 가사 전문 | 소량: 대화 / 대량: 메모장 파일 |
| QT 가이드 | 성경 본문 + 묵상 질문 | 대화 |
| 팀 점수판 | 팀 이름 + 팀 색상 | 대화 |
| 사진 갤러리 | (사진은 나중에 업로드) | 갤러리 구조만 확인 |
| 기도제목 | (학생이 직접 작성) | 게시판 설정만 확인 |
| 종합 앱 | 위 항목 중 포함할 기능 선택 | 대화 |

**콘텐츠 입력 안내 규칙**:
- 10개 이하: 대화로 직접 입력
- 11개 이상: "메모장에 적어서 바탕화면에 저장해 주세요" 안내
- 파일 읽기 후: "이렇게 읽었어요. 맞나요?" 확인
- 오타/빠진 번호: 조용히 수정 후 한 줄로만 보고

**산출물**: 수집된 의도와 콘텐츠를 프로젝트 폴더 내 상태 파일에 저장. 역할 2(코드 생성)가 이 파일을 읽어 코드에 삽입.

### 미리보기 & 피드백 (Phase 5)

- 링크와 함께 핵심 질문 1개만 제시

**사전 테스트 모드** (성찰 C4 신설):
> 사역자가 수련회 전에 2~3명의 학생과 사전 테스트할 수 있는 기능.

```
사역자: "학생 몇 명이랑 먼저 테스트해보고 싶어요"

Claude Code:
  1. 현재 LAN 서버 실행 상태 확인
  2. QR코드 제시: "이 QR코드를 테스트할 학생들에게 보여주세요"
  3. 접속 현황 실시간 표시: "지금 3명이 접속했어요"
  4. 테스트 후 피드백 수집: "학생들이 뭐라고 하나요? 바꿀 부분이 있으면 알려주세요"
  5. 수정 -> 재테스트 반복
```

## 4. 기술 자동화 전략

### Phase 0: 환경 사전 준비 (성찰 B1 전면 개정)

> **핵심 변경**: Phase 0은 "완전 자동"이 아니라 **"기술 도우미가 1회 수행"**으로 명시한다.
> Node.js 설치, 백신 예외 등록, SmartScreen 허용 등은 Claude Code가 자동 제어할 수 없는 Windows 팝업이 개입하기 때문이다.

**Phase 0의 현실적 정의**:

```
Phase 0은 사역자 혼자가 아니라,
"교회 청년부에서 컴퓨터 좀 아는 분"이 1회 도와주는 단계이다.

Claude Code가 안내 메시지를 출력하면,
기술 도우미가 그 안내를 따라 설정을 완료한다.

Phase 0 완료 후, 이후 모든 과정(Phase 1~6)은
사역자가 혼자서 한국어 대화만으로 진행한다.
```

**기술 도우미가 수행하는 것** (1회):

| 항목 | 작업 | Claude Code 역할 |
|------|------|------------------|
| Node.js 설치 | 설치 파일 실행 + UAC 허용 | 설치 확인 안내 출력 |
| 백신 예외 등록 | V3/알약에서 Node.js 허용 | 어떤 프로그램을 허용해야 하는지 안내 |
| SmartScreen 허용 | "실행" 버튼 클릭 | 화면 스크린샷 기반 안내 |
| Claude Code 설치 | 설치 + API 키 입력 | 단계별 안내 |
| Git 설치 (선택) | GitHub Pages 사용 시에만 | 자동 설치 시도, 실패 시 안내 |
| 디스크 공간 확인 | 최소 2GB 여유 확인 | 자동 확인 후 안내 |
| 관리자 권한 확인 | 프로그램 설치 가능 여부 | 자동 확인 후 안내 |

**Claude Code 허용 설정**:
- workflow.md에서 안전한 작업(파일 생성, 서버 실행 등)은 자동 허용되도록 설정 미리 구성
- 사역자가 매번 "허용" 버튼을 누르지 않아도 되게 함

**테크 헬퍼가 없는 경우** (성찰 G5 신설):
```
소규모 교회 등 테크 헬퍼를 구할 수 없는 경우:

  [방법 1] Claude Code가 화면 캡처 기반 단계별 안내를 제공
    -> "이 화면이 보이면 '다음' 버튼을 누르세요" 수준의 안내
    -> 사역자가 혼자 따라할 수 있도록 5분짜리 설치 과정

  [방법 2] 원격 도움 (카카오톡 화면 공유)
    -> 교회 청년부나 지인에게 카카오톡 화면 공유로 원격 안내
    -> "화면 보여주시면 제가 알려드릴게요" 수준

  어느 방법이든, Phase 0 완료 후에는 사역자 혼자 진행 가능.
```

**환경 점검 체크리스트** (Phase 0 완료 기준):

```
[ ] Node.js 설치 완료 (node --version 응답)
[ ] Claude Code 설치 완료 + API 키 인증 성공
[ ] 디스크 공간 2GB 이상
[ ] 백신에서 Node.js 허용됨
[ ] (선택) Git 설치 + GitHub 인증
```

### 환경 자동 감지 (Phase 3 이후, 런타임)

| 항목 | 방법 |
|------|------|
| WiFi IP | `os.networkInterfaces()` -> 192.168.137.x (핫스팟) 우선, 없으면 첫 IPv4 |
| 포트 | 3000부터 순차 시도 -> 3009까지 -> **8080 -> 49152~49162** (성찰 B3: 고포트 추가) |
| 디스크 | 500MB 미만 시 경고 |
| 한글 경로 | `process.cwd()` 감지 -> **바탕화면 또는 문서 폴더** 우선, C:\ 루트는 최후 수단 (성찰 B2) |
| 방화벽 | 포트 접근 테스트 -> 전체 차단 시 **핫스팟 전환 안내** (성찰 E1: 로컬 전용) |

**폴더 생성 위치 우선순위** (성찰 B2):
```
1순위: 바탕화면\church-app\     (사역자가 찾기 쉬움)
2순위: 문서\church-app\         (바탕화면 권한 없을 시)
3순위: C:\church-app\           (위 둘 다 실패 시)
4순위: D:\church-app\           (C드라이브 권한 없을 시)
```

### 네트워크 접속 전략 (성찰 E1 전면 개정 — 터널링 삭제)

> **절대 원칙**: 모든 네트워크 접속은 로컬 범위 내에서만 이루어진다.
> 외부 서버를 경유하는 터널링(ngrok 등)은 사용하지 않는다.
> 학생이 셀룰러(LTE/5G) 데이터를 쓰면 LAN 접속이 불가능하므로,
> **WiFi 연결을 사전 안내하는 것이 핵심 전략이다.**

```
[1순위] 같은 WiFi LAN 접속 (기본)
  학생들이 수련회장 WiFi에 연결 -> 192.168.x.x:PORT 접속
  -> 가장 단순하고 안정적

[2순위] 모바일 핫스팟 (WiFi 문제 시)
  사역자 PC 모바일 핫스팟 ON -> 학생들이 핫스팟에 연결
  -> 최대 약 8대 동시 접속 (팀별 분할 접속)
  -> 핫스팟 IP(192.168.137.x) 자동 감지 -> QR 재생성

[WiFi/핫스팟 둘 다 불가 시]
  -> 실시간 기능은 사용 불가. 솔직히 안내.
  -> 정적 앱(일정표, QT 등)은 GitHub Pages(사역자 동의 시)로 대체 가능
  -> "학생들이 와이파이에 연결해야 앱을 쓸 수 있어요.
     와이파이가 안 되면 일정표/QT 가이드만 따로 만들어드릴게요."

[WiFi 강제 연결 사전 안내] (중요!)
  수련회 시작 전, Claude Code가 자동 생성하는 안내문:
  "앱을 쓰려면 와이파이에 연결해야 해요.
   설정 -> 와이파이 -> [수련회장 WiFi 이름] 선택
   (데이터로는 접속이 안 돼요!)"
  -> A4 인쇄용 HTML로 제공
  -> 수련회 장소에 붙이거나 카카오톡으로 공유
```

### Git 체크포인트 전략

**커밋 시점 & 메시지 포맷**:
```
[초기화] 프로젝트 생성 완료

### 서버 실행 메커니즘 (성찰 E5 전면 신설)

> Claude Code의 Bash 도구는 명령어 실행 후 완료를 기다린다.
> `node server.js`를 foreground로 실행하면 Claude Code가 다음 작업을 못 한다.
> 따라서 서버 실행 방식을 명확히 정의해야 한다.

**서버 백그라운드 실행 전략**:
```
[Phase 6 배포 시]
  Claude Code가 Bash 도구의 run_in_background 옵션을 사용하여
  Node.js 서버를 백그라운드에서 실행한다.

  실행 후:
  - Claude Code는 사역자와 계속 대화 가능
  - 서버는 별도 프로세스로 실행 중
  - 사역자 수정 요청 시 서버 재시작도 Claude Code가 자동 처리

[서버 프로세스 관리] (성찰 G10: PID 파일 대신 포트 기반)
  - PID 파일은 stale 위험이 있으므로 사용하지 않는다
  - 서버 재시작 시: netstat로 해당 포트를 점유한 프로세스 탐색 -> taskkill로 종료 -> 새 서버 시작
  - Claude Code 세션 종료 시: 서버 프로세스는 독립 실행 유지
```

**기술 스택 현실화** (성찰 G7, G8):
```
[데이터 저장] (성찰 G7)
  SQLite는 node-gyp + Python + VS Build Tools가 필요하여
  교회 PC에서 npm install이 실패할 확률이 높다.
  
  -> 순수 JS 솔루션 사용: lowdb (JSON 파일 기반) 또는 단순 JSON 파일
  -> 네이티브 컴파일이 필요한 패키지는 일체 사용하지 않는다
  -> 이 원칙은 workflow.md의 역할 2(코드 생성)에 강제 적용

[번들 크기] (성찰 G8)
  PRD의 300KB 목표는 Socket.io(60KB) + QR 라이브러리(40KB) + 
  한글 폰트(80KB+)를 고려하면 비현실적이다.

  -> 현실적 목표: 500KB 이하 (gzip 후)
  -> 추가 경량화 전략:
     Socket.io 대신 네이티브 WebSocket 사용 (0KB 추가)
     한글 폰트는 시스템 폰트(Malgun Gothic) 우선 사용 (0KB 추가)
     QR 라이브러리는 서버 사이드에서 생성 (클라이언트 번들에 미포함)
  -> 이 전략 적용 시 클라이언트 번들은 앱 코드 + CSS + 서비스워커만으로 구성
```

- **PWA**: 서비스워커 기반 오프라인 캐싱. 모든 앱에 기본 적용
- **한글 폰트**: 시스템 폰트(Malgun Gothic) 우선 사용 (성찰 G8), Pretendard 폴백

**배포 자동 판단 규칙**:
- 실시간 기능 1개라도 포함 -> LAN 서버 배포
- 모두 정적 -> LAN 기본, GitHub Pages 선택 가능

## 7. 디자인 품질 보장 전략 (성찰 A1 + G4: 이 섹션이 디자인 품질의 SOT)

> "중학생이 진짜 앱이라고 느끼는가?" — 이것이 제1 핵심 목적의 핵심이다.
> 자동화 파이프라인이 아무리 완벽해도, 결과물이 촌스러우면 실패다.

> **SOT 선언** (성찰 G4): 디자인 품질 기준은 **이 섹션에만** 정의한다.
> 다른 섹션(역할 3, Phase 4 등)에서는 "→ SS8 참조"로 포인터만 남긴다.

### 디자인 시스템 기본값 (PRD SS9.1에서 가져옴)

```
색상 팔레트 (3가지 세트 — 사역자가 선택):

  [A] 차분한 느낌 (기본):
    --primary: #4F46E5 (인디고)  --secondary: #10B981 (에메랄드)
  [B] 활기찬 느낌:
    --primary: #8B5CF6 (보라)    --secondary: #F59E0B (앰버)
  [C] 따뜻한 느낌:
    --primary: #EC4899 (핑크)    --secondary: #06B6D4 (시안)

타이포그래피:
  Pretendard -> Noto Sans KR -> 시스템 폰트 (폴백 체인)
  본문 16px / 소제목 24px / 빔프로젝터 48px

애니메이션 (필수):
  화면 전환: 250ms ease-out
  버튼 탭: 150ms ease-out
  점수 변경: 300ms 카운트업
  정답: 초록색 플래시 200ms
  오답: 빨간색 흔들림 200ms

모바일 네이티브 느낌 (필수):
  카드 UI (12px radius, 부드러운 그림자)
  상단 헤더 고정
  하단 탭 내비게이션 (종합 앱)
  터치 영역 44x44px 이상
  다크모드 자동 감지
```

### 디자인 품질 자동 검증 (D1~D6)
