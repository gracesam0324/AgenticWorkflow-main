## §0. 이 문서의 정체성

> **이 PRD는 SaaS 제품 기획서가 아니다.**
> **이 PRD는 `workflow.md`를 만들기 위한 사전 요구사항 정의서다.**

```
이 문서의 목적:
  "무엇을 만들 것인가(What)"를 정의하여,
  Claude Code가 workflow.md를 설계할 수 있는 충분한 입력을 제공한다.

구현 주체:
  Claude Code가 사역자의 로컬 PC에서 workflow.md를 따라
  모든 코드를 자동으로 생성·실행·배포한다.
  별도의 개발팀은 없다. 별도의 소프트웨어 패키지도 사전 구축하지 않는다.

작동 방식:
  사역자가 Claude Code와 대화한다.
  → Claude Code가 대화 내용을 바탕으로 앱 코드를 직접 생성한다.
  → 생성된 앱이 사역자의 PC에서 실행된다.
  → 학생들이 같은 와이파이에서 스마트폰으로 접속한다.
```

**핵심 구분**: 별도의 "앱 빌더 시스템"을 먼저 만들고 그것으로 앱을 생성하는 구조가 아니다. Claude Code 자체가 사역자와 대화하면서 앱 코드를 직접 작성하는 구조다.

## §1. 제품 개요

### 1.1 한 문장 정의

> **코딩 지식이 전혀 없는 교회 사역자가 Claude Code와 한국어 대화만으로, 중학생이 실제로 사용할 수 있는 수준의 수련회 앱을 자기 PC에서 자동으로 완성하는 워크플로우.**

### 1.2 배경

#### 교회 사역자의 현실

수련회를 준비하는 사역자(전도사, 교육부장, 청소년부 담당)는 한 명이 모든 것을 동시에 책임진다.

| 책임 영역 | 세부 내용 |
|-----------|-----------|
| 영적 준비 | 설교 준비, QT 자료 제작, 기도회 인도 |
| 행정 | 참가비 수납, 동의서 수거, 버스 예약, 숙소 섭외 |
| 프로그램 | 레크리에이션 기획, 팀 편성, 점수 운영, 시상식 |
| 상담 | 학생 개별 면담, 학부모 소통 |

"수련회 앱을 만들고 싶다"는 바람은 현실적으로 불가능에 가깝다. 외주 개발은 수백만 원, 노코드 툴은 영어 + 가파른 학습 곡선, 엑셀/카카오톡 운영은 실수가 잦고 학생 몰입도가 낮다.

#### 해결 방식

Claude Code는 자연어 지시만으로 실제 동작하는 웹앱을 생성할 수 있다. 본 워크플로우는 Claude Code의 코드 생성 능력과, 사역자가 이미 보유한 능력(한국어로 원하는 것 설명하기) 사이의 모든 기술적 장벽을 자동화로 제거한다.

## §2. 타겟 사용자 & 페르소나

### 2.1 사용자 구조

```
┌─────────────────────────────────────────────┐
│          Primary User (제작자)               │
│        교회 사역자 (전도사/교육부장)           │
│   → Claude Code와 대화, 앱을 만드는 사람     │
└────────────────────┬────────────────────────┘
                     │ 만든 앱을 제공
                     ▼
┌─────────────────────────────────────────────┐
│          Secondary User (사용자)             │
│             중학생 (13~15세)                │
│      → 완성된 앱을 실제로 사용하는 사람      │
└─────────────────────────────────────────────┘
```

### 2.2 페르소나 A — 박지훈 전도사 (29세)

| 항목 | 내용 |
|------|------|
| 직분 | 청소년부 전도사 (사역 2년차) |
| 담당 | 중학생 35명, 고등학생 20명 |
| 기기 | Windows 11 노트북, 삼성 갤럭시 스마트폰 |
| 기술 수준 | 카카오톡, 유튜브, 캔바(Canva) 사용 가능. 엑셀 기초. 코딩 전무 |
| 시간 가용 | 수련회 3주 전부터 기획 시작. 하루 1~2시간 확보 |

**페인 포인트**: 엑셀 점수판 실수, 카카오톡 퀴즈 산만함, 외주 비용 150만 원+

**성공 기준**: 3일 안에 퀴즈 앱과 점수판이 작동. 수련회 당일 기술 문제로 중단 없음.

### 2.3 페르소나 B — 김성은 권사 (48세)

| 항목 | 내용 |
|------|------|
| 직분 | 교육부장 (평신도 봉사자, 15년차) |
| 기기 | Windows 10 데스크톱 (교회 사무실), 아이폰 |
| 기술 수준 | 한글(HWP) 문서, 카카오채널 운영. 코딩 전무 |

**성공 기준**: QR코드 하나로 학생들이 일정 확인. 일정 변경을 대화로 즉시 수정.

### 2.4 페르소나 C — 이다은 청년 간사 (24세)

| 항목 | 내용 |
|------|------|
| 직분 | 청소년부 간사 (자원봉사, 대학교 4학년) |
| 기기 | Windows 11 노트북 (개인), 아이폰 |
| 기술 수준 | 피그마 기초, 노션 능숙. 코딩 전무 |

**성공 기준**: QR 스탬프 랠리 앱 구현. 팀별 실시간 순위 자동 업데이트.

### 2.5 Secondary User — 중학생 (13~15세)

| 항목 | 내용 |
|------|------|
| 기기 | 개인 스마트폰 (안드로이드/iOS 혼재) |
| UI 기대치 | 인스타그램, 카카오톡 수준. "촌스러우면 안 씀" |
| 접근 방식 | QR코드 스캔 → 브라우저에서 바로 열림 (앱 설치 거부감) |

**"실제로 사용한다"의 기준**: QR 스캔 후 빠르게 화면이 뜸. 설명 없이도 직관적. 디자인이 "실제 앱처럼" 보임.

## §3. 핵심 시나리오

> **참고**: 아래 대화에서 "Claude Code:"는 사역자와 직접 대화하는 AI(Claude Code)를 의미한다. 별도의 시스템이나 CLI가 아니다.

### 3.1 시나리오 개요

| # | 앱 유형 | 핵심 기능 | 데이터 동기화 방식 |
|---|---------|-----------|-------------------|
| 1 | 성경 퀴즈 대회 앱 | 팀 대항전, 점수 | 실시간 (진행자→전체) |
| 2 | 팀 미션 스탬프 랠리 | QR 기반 미션 인증 | 개별 (학생→서버) |
| 3 | 수련회 일정표 & 공지 | 일정 보기, 공지 | 정적 (사전 생성) |
| 4 | 찬양 가사 표시 | 빔/폰 동시 지원 | 실시간 (인도자→전체) |
| 5 | QT 가이드 | 매일 다른 본문/질문 | 정적 (사전 생성) |
| 6 | 팀 점수판 & 순위표 | 누적 점수 | 실시간 (관리자→전체) |
| 7 | 사진 공유 갤러리 | 사역자 업로드, 학생 조회 | 단방향 (사역자→학생) |
| 8 | 기도제목 나눔 게시판 | 익명/실명 작성 | 정적 (새로고침) |
| 9 | 종합 수련회 앱 | 위 기능 통합 | 혼합 |

> **유형 범위 규칙 (3차 성찰 반영)**: 위 9가지는 **검증된 유형**이다. 사역자가 다른 유형(예: 출석체크, 투표)을 요청하면 Claude Code가 가능 여부를 판단하고 시도하되, **품질 보장은 검증 유형에 한한다.** 범위 밖 요청의 처리는 §3.7 참고.

### 3.2 시나리오 1 — 성경 퀴즈 대회 앱 (Happy Path)

```
사역자: 이번 수련회에서 성경 퀴즈 대회를 하려고 하는데,
        팀이 4개고 팀마다 학생 8~9명이에요.
        진행자가 문제를 읽으면 학생들이 폰으로 버저를 누르는
        방식으로 하고 싶어요.

Claude Code: 네, 만들 수 있어요! 몇 가지만 확인할게요.
             퀴즈 문제는 미리 입력해 두실 건가요,
             아니면 당일에 직접 진행하실 건가요?

사역자: 문제는 미리 50개 입력할게요.

Claude Code: 알겠어요! 이렇게 만들게요.
             [진행자 화면] 문제 넘기기, 정답 공개, 점수 추가
             [팀 버저 화면] 큼직한 버저 버튼
             [스코어보드] 4팀 점수 실시간 표시 (빔프로젝터용)
             시작할까요?
```

### 3.3 시나리오 — 사역자가 모호한 요청을 했을 때 (Unhappy Path)

```
사역자: 뭔가 재밌는 앱을 만들고 싶어요.

Claude Code: 좋아요! 어떤 상황에서 쓸 앱인지 알려주시면
             더 잘 도와드릴 수 있어요.
             예를 들어:
             1) 팀끼리 경쟁하는 퀴즈 대회
             2) 캠프장 곳곳에서 미션을 수행하는 스탬프 랠리
             3) 수련회 일정을 학생들이 폰에서 보는 일정표
             어떤 게 제일 가까우세요?

사역자: 음... 1번이랑 3번 둘 다요.

Claude Code: 두 가지를 하나의 앱에 넣을 수 있어요!
             일정표 탭과 퀴즈 탭이 있는 종합 앱을 만들게요.
             먼저 팀 구성부터 알려주시겠어요? 팀이 몇 개인가요?
```

### 3.4 시나리오 — 수정 요청 & 롤백

```
사역자: 아까 만든 퀴즈 앱 배경색을 파란색으로 바꿔주세요.

Claude Code: 바꿨어요! 브라우저에서 확인해 보세요.

사역자: 음... 아까 것이 더 나았어요. 되돌릴 수 있어요?

Claude Code: 네! 이전 상태로 돌려놨어요. 확인해 보세요.
```

### 3.5 시나리오 — 기술 오류가 발생했을 때

```
[내부적으로 npm install 실패 발생]

Claude Code: 앱을 만드는 중에 작은 문제가 생겼는데,
             제가 알아서 해결하고 있어요. 잠시만 기다려 주세요.
             ... (자동 재시도)
             해결됐어요! 계속 진행할게요.
```

### 3.6 나머지 시나리오 (요약)

- **스탬프 랠리**: 미션 장소 8곳 + QR코드 자동 생성 + PDF 출력
- **일정표 & 공지**: 자연어로 일정 입력 → 모바일 앱 자동 생성
- **찬양 가사**: 인도자 화면 + 빔 화면 + 학생 폰 동시 표시
- **QT 가이드**: 3일치 성경 본문, 묵상 질문, 기도제목
- **팀 점수판**: 관리자(비밀번호 보호)가 점수 입력 → 전체 화면 반영
- **사진 갤러리**: 사역자가 사진 업로드 → 학생들이 조회 + 슬라이드쇼
- **기도제목 게시판**: 익명/실명 선택, "함께 기도할게요" 반응 버튼
- **종합 수련회 앱**: 위 기능 통합, QR코드 하나로 접근

### 3.7 시나리오 — 범위 밖 요청 (3차 성찰 반영)

```
사역자: 유튜브처럼 영상을 올리고 학생들이 보는 앱을 만들고 싶어요.

Claude Code: 영상 앱은 저장 공간이 많이 필요하고
             인터넷이 느린 수련회장에서는 끊길 수 있어요.
             대신 이런 방법은 어떨까요?
             1) 유튜브에 올린 영상 링크를 앱에 모아두기
             2) 사진 갤러리 앱으로 사진과 짧은 글 공유하기
             어떤 게 좋으실까요?
```

```
사역자: 카카오톡 같은 채팅 앱을 만들어주세요.

Claude Code: 실시간 채팅 앱은 만들 수 있지만,
             중학생들이 익명으로 채팅하면
             관리가 어려울 수 있어요.
             대신 이런 기능은 어떨까요?
             1) 기도제목 나눔 게시판 (글로 소통)
             2) 팀별 공지 게시판
             어떤 게 필요하신 건지 더 알려주실래요?
```

**범위 밖 요청 처리 원칙**:
- "못 만들어요"로 거절하지 않는다
- 왜 어려운지 **사역자가 이해할 수 있는 이유**를 설명한다
- 반드시 **대안**을 제시한다
- 대안도 §3.1의 검증된 유형 범위 내에서 제안한다

## §5. UX 설계 원칙

### 5.1 원칙 1 — 대화 중심 (Conversational First)

> **사역자는 Claude Code와 오직 자연어 대화로만 소통한다. 별도의 CLI 명령어, 설정 화면, 폼은 없다.**

```
❌ "프레임워크를 선택하세요: React / Vue"
❌ 설정 파일(.yaml, .json)을 사역자가 편집
❌ 터미널에 명령어 입력 (camp-builder init 같은 별도 CLI 없음)
❌ [Y/n] 같은 프롬프트

✅ "어떤 수련회 프로그램이 필요하신가요?"
✅ 모든 기술적 결정은 Claude Code가 자동 수행
✅ "이렇게 만들었어요, 확인해 보시겠어요?"
```

**대화 설계 규칙**

| 규칙 | 적용 |
|------|------|
| 질문은 한 번에 최대 2개 | 가장 중요한 것만 먼저 |
| 예시를 항상 제시 | 구체적 예시로 선택 유도 |
| 선택지는 2~3개 이내 | 과다 선택지는 혼란 |
| 진행 상황 표시 | "지금 3단계 중 2단계예요" |
| 모호한 입력 시 재질문 | 추측하지 않고 확인 (§3.3 참고) |
| 기술 오류 시 자동 해결 | 사역자에게 에러를 보여주지 않음 (§3.5 참고) |

### 5.2 원칙 2 — 사전 설치는 인정하되, 최소화

> **Claude Code 실행을 위한 사전 설치는 불가피하다. 이를 숨기지 않고, 가장 단순한 경로를 안내한다.**

```
사전 설치 필요 항목 (§10.1에서 상세 안내):
  1. Node.js (Claude Code 실행을 위해 필요)
  2. Claude Code 자체

이후 모든 과정 (앱 생성에 필요한 프레임워크, 라이브러리 등):
  → Claude Code가 자동으로 설치하고 구성
  → 사역자에게 기술적 과정을 노출하지 않음
```

### 5.3 원칙 3 — 미리보기 필수 (Preview Before Build)

> **앱 생성 전 반드시 구조/디자인 미리보기를 제공하고 사역자 확인을 받는다.**

1. **텍스트 구조도** (즉시) → "이런 구조로 만들 예정이에요"
2. **실제 화면** (생성 후) → 브라우저 자동 열림, 사역자가 직접 확인
3. **수정 반복** → 대화로 수정 요청, 즉시 반영

### 5.4 원칙 4 — 즉시 테스트 (Instant Preview)

> **앱이 생성되면 즉시 사역자의 브라우저에서 확인 가능해야 한다.**

Claude Code가 자동으로: 로컬 개발 서버 실행 → 브라우저 자동 오픈 → 같은 와이파이의 스마트폰용 QR코드 표시

### 5.5 원칙 5 — 반복 수정 (Easy Iteration)

> **완성된 앱의 수정은 대화로. "배경색을 파란색으로" → 즉시 반영.**

지원하는 수정: 색상/디자인, 콘텐츠 변경, 기능 추가/제거, 구조 변경, 이전 버전으로 롤백

**시각적 선택지 제시 원칙 (3차 성찰 반영)**:
- 색상 변경 시: **3가지 후보**를 나란히 보여주고 "1번, 2번, 3번 중 어떤 게 좋으세요?" 방식으로 선택 유도
- "좀 더 밝게" 같은 모호한 수정: 즉시 반영 후 "이 정도면 될까요? 더 밝게 할까요?" 확인
- 3회 이상 같은 항목 수정 반복 시: "혹시 이런 느낌이세요?" + 구체적 이미지/색상 예시 제시
- 텍스트 대화만으로 맞추기 어려운 것(색상, 레이아웃)은 **즉시 반영 + 확인** 방식 사용

### 5.6 원칙 6 — 앱 유형별 최적 배포

> **배포 전략은 앱의 특성(정적/실시간)에 따라 자동으로 결정된다. 사역자에게 기술적 선택을 요구하지 않는다.**

```
┌─────────────────────────────────────────────────────────────────┐
│  앱 유형에 따른 배포 전략 (Claude Code가 자동 판단)             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [A] 정적 앱 → LAN 서버 배포 (기본, 1순위)                     │
│      또는 GitHub Pages (선택 옵션 — §7.7 설정 완료 시)         │
│      대상: 일정표, QT 가이드, 찬양 가사, 사진 갤러리, 공지     │
│      기본: LAN 서버로 배포 (가장 단순, 추가 설정 불필요)        │
│      선택: GitHub Pages (§7.7 설정 완료 시, PC 꺼도 접속 가능)  │
│      QR코드: 로컬 IP 또는 GitHub URL 기반 자동 생성             │
│                                                                 │
│  [B] 실시간 앱 → LAN 서버 배포 (사역자 PC)                     │
│      대상: 성경 퀴즈(버저), 팀 점수판, 실시간 동기화 기능       │
│      방식: 사역자 PC에서 서버 실행 → 같은 와이파이 접속        │
│      QR코드: 로컬 IP(192.168.x.x) 기반 자동 생성               │
│      주의: 사역자 PC가 켜져 있어야 함                           │
│                                                                 │
│  [C] 종합 앱 → LAN 서버 단일 진입점 (§6.8 참고)               │
│      정적+실시간 모두 LAN 서버에서 서빙                         │
│      학생에게는 QR코드 하나로 통합 진입                         │
│                                                                 │
│  [D] LAN 폴백 — 와이파이 클라이언트 격리 시                    │
│      사역자 PC의 모바일 핫스팟 활성화 안내                      │
│      ⚠ 핫스팟 최대 동시 접속: 약 8대 (팀별 분할 접속 필요)     │
│      Claude Code가 핫스팟 IP 자동 감지 → QR코드 재생성         │
└─────────────────────────────────────────────────────────────────┘
```

**핵심**: Claude Code가 사역자의 요청을 분석하여 정적/실시간을 자동 판단하고, 최적의 배포 경로를 선택한다. 사역자에게는 "앱이 완성됐어요! 이 QR코드를 학생들에게 공유하세요"만 전달한다.

### 5.7 원칙 7 — 안전한 실패 (Safe Failure)

> **기술적 에러는 Claude Code가 자동 처리하거나, 사역자 언어로 번역하여 전달한다.**

- 기술적 에러 메시지를 사역자에게 직접 보여주지 않음
- 자동 복구 가능한 에러는 Claude Code가 알아서 해결
- 복구 불가능한 에러만 사역자에게 안내하되, 해결 방법을 함께 제시
- 모든 변경 전 Git 체크포인트 → 언제든 이전 상태로 롤백

## §6. 핵심 기능 명세

### 6.1 Claude Code 실행 파이프라인

```
사역자가 Claude Code에게 말한다
     │
     ▼
[1] 의도 파악
     │  Claude Code가 대화에서 앱 유형 감지, 필요 정보 추출
     │  부족한 정보는 재질문 (§3.3 패턴)
     ▼
[2] 구조 미리보기
     │  텍스트 기반 앱 구조를 보여주고 사역자 확인
     ▼
[3] 코드 자동 생성
     │  Claude Code가 직접 웹앱 코드를 작성
     │  (프레임워크, 스타일, 로직 모두 포함)
     ▼
[4] 로컬 서버 실행 & 미리보기
     │  localhost 서버 자동 시작, 브라우저 자동 열기
     │  같은 와이파이 접속용 QR코드 표시
     ▼
[5] 수정 반복
     │  사역자 피드백 → Claude Code가 코드 수정 → 즉시 반영
     │  "아까 것이 더 좋았어요" → Git 롤백
     ▼
[6] 최종 배포
       LAN QR코드 생성 (기본)
       인쇄용 PDF 출력
       (선택) 외부 배포
```

### 6.2 앱 유형별 핵심 기능

Claude Code가 사역자의 요구에 따라 아래 기능들을 조합하여 앱을 생성한다. 사전 제작된 템플릿이 아니라, Claude Code가 대화 내용을 바탕으로 코드를 직접 작성한다.

| 앱 유형 | 핵심 화면 구성 | 데이터 동기화 |
|---------|---------------|--------------|
| **성경 퀴즈** | 진행자 콘솔 + 팀 버저 + 스코어보드(빔용) | 실시간 |
| **스탬프 랠리** | QR 스캔 화면 + 미션 표시 + 팀 스탬프 현황 | 개별 → 서버 |
| **일정표** | 날짜별 탭 + 현재 일정 하이라이트 + 공지 | 정적 |
| **찬양 가사** | 인도자 제어 + 빔 대형 가사 + 학생 폰 가사 | 실시간 |
| **QT 가이드** | 날짜별 본문 + 묵상 질문 + 메모 기능 | 정적 + 로컬 저장 |
| **팀 점수판** | 관리자 점수 입력 + 전체 순위 표시(빔용) | 실시간 |
| **사진 갤러리** | 사역자 업로드 + 학생 갤러리 조회 + 슬라이드쇼 | 단방향 |
| **기도제목** | 작성(익명/실명) + 목록 조회 + 반응 버튼 | 새로고침 |
| **종합 앱** | 위 기능 통합 + 탭 내비게이션 + 팀 선택 | 혼합 |

### 6.3 데이터 동기화 전략

> **성찰 반영**: WebSocket이 교회 환경(방화벽, 클라이언트 격리)에서 실패할 수 있으므로, 폴링 폴백을 기본 내장한다.

| 전략 | 방식 | 적용 대상 |
|------|------|-----------|
| **실시간 (1순위)** | WebSocket (Socket.io) | 퀴즈, 점수판, 가사 |
| **실시간 폴백 (자동)** | HTTP 폴링 (5초 간격) | WebSocket 연결 실패 시 자동 전환 |
| **정적** | 사전 생성된 HTML/JSON → **GitHub Pages 배포 가능** | 일정표, QT, 공지 |
| **단방향** | 사역자 업로드 → 학생 조회 → **GitHub Pages 배포 가능** | 사진 갤러리 |

WebSocket 실패 시 사역자에게 "실시간 연결이 불안정해서 5초마다 자동 새로고침 방식으로 전환했어요"라고 안내한다.

**브로드캐스트 방향 제한 (3차 성찰 반영)**:
> 35명 이상 동시 접속 시 성능을 보장하기 위해, 실시간 데이터 흐름을 **단방향**으로 제한한다.

```
진행자/관리자 → 서버 → 전체 학생  (1:N 브로드캐스트)
학생 개인     → 서버              (1:1 개별 전송, 브로드캐스트 아님)

예: 퀴즈 버저
  학생이 버저 누름 → 서버에만 전송 (다른 학생에게 전파하지 않음)
  진행자가 정답 공개 → 서버 → 전체 학생에게 브로드캐스트
  
이렇게 하면 50명이 동시에 버저를 눌러도 서버 부하는 50건(1:1),
브로드캐스트는 진행자 1건(1:N)으로 제한됨.
```

### 6.4 커스터마이징 범위

Claude Code가 대화를 통해 자동으로 반영하는 항목:

- **교회/수련회 정보**: 이름, 날짜, 장소, 주제 구절
- **팀 구성**: 팀 수, 팀 이름, 팀 색상
- **디자인**: 메인 색상, 폰트 스타일, 로고 (사역자가 파일 경로 알려주면 적용)
- **콘텐츠**: 퀴즈 문제, 일정, 찬양 가사, 미션 내용, QT 본문
- **기능**: 다크모드, 애니메이션, 사운드 효과

### 6.5 URL 구조 (실시간 기능 포함 앱)

```
/           → 참가자 메인 화면 (학생)
/admin      → 진행자 콘솔 (비밀번호 보호, 사역자/교사)
/screen     → 빔프로젝터 대형 화면
```

### 6.6 QR 코드 & 배포 산출물 (3차 성찰 보강)

Claude Code가 배포 단계에서 자동 생성하는 것들:

- **QR 코드 이미지** (PNG) — 로컬 IP 또는 GitHub Pages URL 기반
- **인쇄용 HTML 페이지** — A4 1페이지에 QR + 수련회 이름 + 접속 안내문. `window.print()`로 브라우저에서 바로 인쇄 (별도 PDF 라이브러리 불필요 — 경량 빌드 원칙 준수)
- **팀별 QR** (스탬프 랠리 시) — 미션 장소별 고유 QR + 장소명

**QR코드 전달 방법 3가지** (프린터 없는 경우 대비):

| 방법 | 사용 시점 | Claude Code 안내 |
|------|-----------|------------------|
| **1. 인쇄** | 프린터 있을 때 | "브라우저에서 인쇄 버튼(Ctrl+P)을 누르세요" |
| **2. 빔프로젝터** | 현장에 빔 있을 때 | "이 화면을 빔프로젝터에 띄워주세요" |
| **3. 카카오톡 전송** | 둘 다 없을 때 | "QR코드 이미지를 저장해서 카카오톡 단체방에 보내주세요" + QR 이미지 파일 위치 안내 |

### 6.7 콘텐츠 관리

- **성경 구절**: 사역자가 직접 입력하거나, Claude Code에게 "요한복음 3장 16절 넣어줘"라고 요청하면 AI가 해당 구절을 작성 (§4.8 저작권 정책 준수)
- **퀴즈 문제**: Claude Code가 사역자가 지정한 성경 범위를 바탕으로 AI 생성. 사역자 확인 후 확정.
- **일정표**: 자연어 입력 → Claude Code가 구조화

#### 6.7.1 콘텐츠 입력 방식 (2차 성찰 반영)

> 사역자가 콘텐츠를 입력하는 방법을 콘텐츠 양에 따라 구분한다. Claude Code가 자동으로 최적 방식을 안내한다.

| 콘텐츠 유형 | 분량 | 입력 방식 | 예시 |
|-------------|------|-----------|------|
| **퀴즈 문제** (소량) | 10문제 이하 | 대화로 직접 입력 | "1번: 노아 방주에 탄 사람은? 정답: 노아 가족 8명" |
| **퀴즈 문제** (대량) | 11문제 이상 | 텍스트 파일 또는 엑셀 | Claude Code가 "문제를 메모장에 적어서 저장해 주세요. 형식은 이렇게요: ..." 안내 |
| **일정표** | 전체 | 대화로 자연어 입력 | "첫째 날 9시 개회예배, 10시 팀 게임, 12시 점심..." |
| **찬양 가사** | 곡당 | 대화 또는 파일 | 가사가 길면 "메모장에 붙여넣어 주세요" 안내 |
| **팀 이름/정보** | 전체 | 대화로 직접 | "1팀 사자팀 빨간색, 2팀 독수리팀 파란색..." |
| **성경 구절** | 구절당 | 대화로 요청 | "요한복음 3:16 넣어줘" |

**파일 입력 안내 규칙**:
- 사역자에게 파일 형식을 지정할 때 **메모장(텍스트 파일)** 기준으로 안내 (엑셀보다 단순)
- 파일 위치는 "바탕화면에 저장해 주세요"로 통일
- Claude Code가 파일 읽기 후 "이렇게 읽었어요. 맞나요?" 확인

### 6.8 종합 앱 통합 아키텍처 (2차 성찰 반영)

> **문제**: 종합 수련회 앱은 정적 기능(일정, QT)과 실시간 기능(퀴즈, 점수판)을 동시에 포함한다. GitHub Pages(정적)와 LAN 서버(실시간)를 어떻게 하나의 앱으로 통합하는가?

**해결 방식: LAN 서버 단일 진입점**

```
종합 앱 = LAN 서버가 모든 것을 서빙한다.

이유:
  - 정적 콘텐츠(일정, QT)도 LAN 서버에서 서빙 가능
  - 실시간 기능은 LAN 서버가 필수
  - 따라서 종합 앱은 LAN 서버 하나에서 모든 기능을 통합
  - QR코드 하나 → 하나의 진입점 → 탭/메뉴로 기능 전환
  - GitHub Pages는 종합 앱에서는 사용하지 않음

종합 앱 구조:
  사역자 PC (Node.js 서버)
    ├── / (메인) → 탭 내비게이션
    │     ├── 일정 탭 (정적 콘텐츠)
    │     ├── 퀴즈 탭 (실시간)
    │     ├── 점수판 탭 (실시간)
    │     ├── 가사 탭 (실시간)
    │     ├── QT 탭 (정적)
    │     └── 기도제목 탭 (정적 + 새로고침)
    ├── /admin → 관리자 콘솔
    └── /screen → 빔프로젝터 화면

GitHub Pages는 언제 사용?
  - 단독 정적 앱만 만들 때 (예: 일정표만, QT만)
  - 수련회 끝난 후 기록 보존용 정적 사이트로 내보내기 (선택)
```

**배포 자동 판단 규칙**:

```
Claude Code 내부 판단 로직:

  사역자 요청 분석
    ├── 실시간 기능이 하나라도 포함? ──YES──→ LAN 서버 배포
    │                                          (정적 콘텐츠도 LAN에서 서빙)
    │
    └── 모두 정적 콘텐츠? ──YES──→ GitHub Pages 배포
                                    (사역자 PC 꺼도 접속 가능)
```

## §7. 기술 방향

> **주의**: 이 섹션은 Claude Code가 workflow.md를 작성할 때 참고할 기술적 방향을 제시한다. 구체적인 파일 구조, 코드, 라이브러리 버전은 Claude Code가 구현 시점에 최적의 선택을 한다.

### 7.1 생성되는 앱의 기술 방향

> **성찰 반영**: 기술 스택을 앱 유형(정적/실시간)에 따라 분리한다. GitHub Pages 배포와의 정합성을 확보한다.

**[A] 정적 앱 (GitHub Pages 배포)**

| 영역 | 방향 | 이유 |
|------|------|------|
| **기반** | 순수 HTML + CSS + JavaScript | GitHub Pages는 정적 호스팅 — 서버 코드 불가. 빌드 도구 의존성 최소화로 안정성 확보 |
| **스타일링** | CSS (커스텀 또는 경량 프레임워크) | 번들 크기 최소화, CDN 의존성 제거 |
| **오프라인** | PWA Service Worker | 수련회 현장 인터넷 불안정 대비 |
| **배포** | GitHub Pages (자동) | 사역자 PC 꺼도 학생 접속 가능 |
| **한국어** | Noto Sans KR 서브셋 또는 시스템 폰트 | 한글 렌더링 + 경량화 |

**[B] 실시간 앱 (LAN 서버 배포)**

| 영역 | 방향 | 이유 |
|------|------|------|
| **서버** | Node.js + Express | 사역자 PC에서 로컬 서버로 실행 |
| **실시간** | Socket.io + HTTP 폴링 폴백 | 교회 와이파이 환경 안정성 |
| **프론트엔드** | HTML + CSS + JS (서버사이드 렌더링 불필요) | 경량, 빠른 로딩 |
| **데이터** | SQLite 또는 JSON 파일 | 사역자 PC 로컬 저장 |
| **배포** | 사역자 PC 로컬 서버 | 같은 와이파이 내 LAN 접속 |

**프레임워크 선택 원칙**: Claude Code가 앱 복잡도를 판단하여 최적의 기술을 자동 선택한다. 단순한 앱에 무거운 프레임워크를 사용하지 않는다. 사역자에게 기술 선택을 묻지 않는다.

### 7.2 Claude Code의 역할

```
Claude Code는 다음을 직접 수행한다:
  ✓ 사역자와 대화하여 요구사항 파악
  ✓ 앱 코드 전체를 직접 작성 (HTML, CSS, JS, 서버 코드)
  ✓ 필요한 패키지 자동 설치 (npm install 등)
  ✓ 로컬 개발 서버 실행
  ✓ QR 코드 생성
  ✓ Git 체크포인트 관리
  ✓ 에러 자동 처리 및 복구

Claude Code는 다음을 하지 않는다:
  ✗ 별도의 "앱 빌더 시스템"을 먼저 구축
  ✗ 사역자에게 터미널 명령어 입력을 요구
  ✗ 기술적 에러를 사역자에게 직접 노출
```

### 7.3 코드 생성 전략 (2차 성찰 반영)

> Claude Code가 앱 코드를 어떤 순서와 방식으로 생성하는지 정의한다.

**생성 순서**: 구조 → 콘텐츠 → 스타일 → 기능 → 검증

```
[1] 프로젝트 뼈대 생성
    폴더 구조 + 메인 HTML + 기본 CSS + 빈 JS
    → Git 체크포인트: "프로젝트 초기화"

[2] 콘텐츠 삽입
    퀴즈 문제, 일정, 성경 구절 등 사역자가 제공한 콘텐츠
    → Git 체크포인트: "콘텐츠 삽입 완료"

[3] 스타일링 & 디자인
    색상, 폰트, 레이아웃, 애니메이션, 다크모드
    → Git 체크포인트: "디자인 적용"

[4] 기능 구현
    실시간 동기화, 관리자 콘솔, PWA 설정
    → Git 체크포인트: "기능 구현 완료"

[5] 품질 검증 (§9.2)
    자동 검증 실행 → 문제 발견 시 자동 수정 → 재검증
    → Git 체크포인트: "검증 완료 — 배포 준비"
```

**수정 요청 시 전략**:

| 수정 유형 | 방식 | 예시 |
|-----------|------|------|
| **스타일 변경** | 해당 CSS/HTML만 부분 수정 | "배경색 파란색으로" → CSS 변수 1줄 변경 |
| **콘텐츠 변경** | 해당 데이터만 부분 수정 | "3번 문제 수정" → 데이터 파일 1줄 변경 |
| **기능 추가** | 관련 파일만 추가/수정 | "타이머 추가" → JS 모듈 추가 + HTML 수정 |
| **구조 변경** | 영향 범위 분석 후 재생성 | "탭 구조를 카드로" → 레이아웃 전체 재생성 |
| **롤백** | Git 체크포인트로 복원 | "아까 것이 더 좋았어요" → git checkout |

**원칙**: 사역자의 수정 요청은 **최소 범위 변경**으로 처리한다. "색상 변경"에 전체 코드를 재생성하지 않는다.

### 7.4 경량 빌드 전략

교회 와이파이(5Mbps 이하)에서 50명이 동시 접속하는 현실을 고려:

| 기준 | 목표 |
|------|------|
| **앱 번들 크기** | 300KB 이하 (gzip 압축 후) |
| **이미지** | WebP 포맷, 지연 로딩 |
| **폰트** | 서브셋 한국어 폰트 (전체 NotoSansKR 대신 자주 쓰는 글자만) |
| **초기 로딩** | 최소 HTML + CSS 먼저 → JS 지연 로딩 |

5Mbps에서 50명 접속 시 1인당 약 100Kbps. 300KB 번들은 약 24초가 걸리므로, **PWA 캐싱이 필수** — 최초 접속 후 오프라인 캐시에서 즉시 로딩.

### 7.5 AI 모델 사용

Claude Code 자체가 Claude API를 사용하므로, 별도의 LLM 폴백 체인은 불필요하다. Claude Code의 API 연결이 끊기면 앱 생성 자체가 불가능하며, 이를 사역자에게 솔직히 안내한다.

> "인터넷 연결이 끊어져서 지금은 앱을 만들 수 없어요. 인터넷이 다시 연결되면 알려주세요."

### 7.6 Windows 환경 특수 처리 (2차 성찰 반영)

> 사역자의 PC는 Windows 10/11이다. Windows 환경에서 발생하는 특수한 기술적 상황을 Claude Code가 자동으로 처리해야 한다.

| 상황 | 문제 | Claude Code 자동 처리 |
|------|------|----------------------|
| **방화벽 팝업** | Node.js 서버 실행 시 Windows 방화벽이 "이 앱의 네트워크 접근을 허용하시겠습니까?" 팝업 표시 | 사역자에게 한국어로 안내: "화면에 '허용' 버튼이 뜨면 눌러주세요. 학생들이 앱에 접속하려면 필요해요." |
| **포트 충돌** | 3000번 포트가 이미 사용 중 | 자동으로 다음 포트(3001, 3002...) 시도. 사역자에게는 "준비 중이에요" 정도만 표시 |
| **로컬 IP 감지** | LAN QR코드 생성을 위해 사역자 PC의 와이파이 IP 필요 | Node.js `os.networkInterfaces()`로 와이파이 어댑터의 IPv4 주소 자동 감지. Wi-Fi / Ethernet 인터페이스 우선 |
| **브라우저 자동 열기** | 앱 미리보기 시 브라우저 자동 실행 | Windows에서 `start http://localhost:PORT` 명령 사용 |
| **절전 모드 방지** | 수련회 중 노트북이 절전 모드에 들어가면 서버 중단 | Claude Code가 Windows 전원 설정 변경을 안내: "노트북 덮개를 닫지 마시고, 전원 설정에서 절전을 '안 함'으로 바꿔주세요" |
| **경로 구분자** | Windows는 `\`, Node.js는 `/` | Claude Code가 생성하는 코드에서 `path.join()` 사용으로 자동 처리 |
| **한글 경로** | 사역자 PC 사용자명이 한글일 수 있음 (예: `C:\Users\김전도사\`) | 프로젝트 폴더를 `C:\church-app\` 등 영문 경로에 생성하여 인코딩 문제 회피 |
| **모바일 핫스팟 IP** | 핫스팟 모드에서는 IP 대역이 다름 (보통 192.168.137.x) | `os.networkInterfaces()`에서 핫스팟 어댑터 자동 감지 → QR코드 재생성 |

### 7.7 GitHub Pages 자동화 전체 흐름 (2차 성찰 반영)

> GitHub Pages 배포는 사역자가 기술적 절차를 이해할 필요 없이, Claude Code가 전체 과정을 자동 수행한다. 단, 최초 1회 인증 설정은 사역자 참여가 필요하다.

**최초 설정 (1회만 — Step 0에서 수행)**:

```
Claude Code가 단계별로 안내하는 것:

[1] Git 설치 확인
    → Claude Code가 `git --version` 실행
    → 미설치 시: "프로그램 하나를 설치해야 해요. 
      이 링크를 열어서 '다운로드' 버튼을 눌러주세요."
    → 설치 후 확인

[2] GitHub CLI(gh) 설치
    → Claude Code가 `gh --version` 실행
    → 미설치 시: winget으로 자동 설치 시도
      `winget install --id GitHub.cli`
    → winget 실패 시: 다운로드 링크 안내

[3] GitHub 인증
    → `gh auth login` 실행
    → "브라우저 창이 열리면, GitHub에 로그인해 주세요" 안내
    → 인증 완료 확인: `gh auth status`

[4] 인증 성공 확인
    → "GitHub 준비가 끝났어요! 이제 앱을 만들면 
      자동으로 인터넷에 올려드릴게요."
```

**배포 시 (매번 자동)**:

```
Claude Code가 자동 실행:

[1] 저장소 생성 (최초) 또는 기존 저장소 사용
    → `gh repo create church-camp-app --public --source=. --push`
    → 이미 존재하면 `git push`만 실행

[2] GitHub Pages 활성화
    → `gh api repos/{owner}/{repo}/pages -X POST 
       -f source.branch=main -f source.path=/`
    → 이미 활성화되어 있으면 건너뜀

[3] 배포 완료 대기
    → `gh api repos/{owner}/{repo}/pages` 으로 상태 확인
    → 배포 완료까지 보통 30초~2분

[4] URL 확인 & QR 생성
    → `https://{username}.github.io/{repo-name}/`
    → 해당 URL로 QR코드 생성 + 인쇄용 PDF 생성

[5] 사역자에게 안내
    → "앱이 완성됐어요! 이 주소로 접속하면 돼요: [URL]
       QR코드도 만들었어요. 인쇄해서 붙여주세요.
       이 주소는 노트북을 꺼도 계속 접속할 수 있어요."
```

**에러 처리**:

| 에러 | 원인 | 자동 대응 |
|------|------|-----------|
| `gh auth` 실패 | 브라우저에서 인증 안 함 | "브라우저에서 로그인을 완료해 주세요" 재안내 |
| `gh repo create` 실패 | 이미 같은 이름 존재 | 자동으로 이름 뒤에 날짜 추가 (예: `camp-app-2026`) |
| `git push` 실패 | 네트워크 문제 | 3회 재시도 후 "인터넷 연결을 확인해 주세요" 안내 |
| Pages 활성화 실패 | API 제한 | 수동 안내: "GitHub 사이트에서 Settings → Pages를 열어주세요" |
| 빌드 실패 | 파일 크기 초과 등 | 이미지 최적화 후 재시도 |

## §8. 안전 & 보안 원칙

### 8.1 코드 생성 안전

Claude Code가 생성하는 모든 앱 코드는 다음 원칙을 준수한다:

| 원칙 | 내용 |
|------|------|
| **외부 전송 금지** | 생성된 앱이 사역자 PC 외부로 데이터를 전송하지 않음 (명시적 배포 제외) |
| **사용자 입력 무해화** | 학생이 입력하는 닉네임, 기도제목 등에 스크립트 인젝션 방지 |
| **외부 스크립트 제한** | CDN 폰트(Google Fonts) 외 외부 스크립트 로드 금지 |
| **파일 접근 범위** | 프로젝트 폴더 내부로만 제한 |
| **관리자 보호** | 진행자 콘솔(/admin)은 비밀번호 보호. 비밀번호는 사역자가 대화로 설정 |

### 8.2 콘텐츠 안전 (교회 환경)

Claude Code가 앱 콘텐츠 생성 시 준수하는 정책:

**항상 허용**: 성경 구절, 찬양/예배 콘텐츠, 교육 적합 게임 요소

**항상 차단**: 폭력적/선정적 콘텐츠, 타 종교 비방, 비속어, 도박/사행성 요소

**퀴즈 생성 정책**: 성경 내용에 근거한 문제만 생성. 답이 성경에서 확인 가능. 신학적 논쟁 주제 회피.

### 8.3 사진 갤러리 정책

| 항목 | 정책 |
|------|------|
| **업로드 방향** | 사역자만 업로드 가능 (단방향). 학생은 조회만 |
| **이유** | 학생 상호 업로드 시 부적절 이미지 위험 + 개인정보 노출 위험 |
| **용량 제한** | 이미지당 최대 5MB, 자동 리사이징 적용 |
| **총 용량** | 최대 500MB (약 100~200장) |
| **슬라이드쇼** | 사역자가 업로드한 사진만으로 구성 |

### 8.4 에러 처리 결정 트리 (2차 성찰 반영)

> Claude Code가 앱 생성/실행 중 발생하는 에러를 어떻게 처리하는지 정의한다. 핵심 원칙: **사역자에게 기술적 에러를 보여주지 않되, 복구 불가능한 상황은 솔직히 안내한다.**

```
에러 발생
  │
  ├─ [자동 복구 가능?]
  │     │
  │     ├─ YES → 자동 복구 시도 (최대 3회)
  │     │          ├─ 성공 → 사역자에게 알리지 않고 계속 진행
  │     │          └─ 실패 → [사역자 개입 필요?]로 이동
  │     │
  │     └─ NO → [사역자 개입 필요?]
  │               │
  │               ├─ 사역자가 할 수 있는 것 → 한국어 안내
  │               │    "와이파이 연결을 확인해 주세요"
  │               │    "방화벽 허용 버튼을 눌러주세요"
  │               │
  │               └─ 사역자가 할 수 없는 것 → 솔직히 안내 + 대안 제시
  │                    "지금은 이 기능을 만들 수 없어요.
  │                     대신 이렇게 하면 비슷한 결과를 얻을 수 있어요."
  │
  └─ [데이터 손실 위험?]
        │
        ├─ YES → 즉시 중단 + Git 체크포인트로 롤백 + 사역자 안내
        └─ NO → 위 흐름 진행
```

**에러 유형별 처리 규칙**:

| 에러 유형 | 예시 | 자동 복구 | 최대 재시도 | 사역자 안내 |
|-----------|------|-----------|-------------|-------------|
| **패키지 설치** | `npm install` 실패 | 캐시 삭제 후 재시도 | 3회 | 3회 실패 시 "인터넷 연결 확인" 안내 |
| **포트 충돌** | `EADDRINUSE` | 다음 포트 자동 시도 | 10회 (3000~3009) | 10회 실패 시 "다른 프로그램을 닫아보세요" |
| **Git 충돌** | merge conflict | 자동 해결 시도 | 1회 | 실패 시 새 브랜치에서 재시작 |
| **API 연결** | Claude API 타임아웃 | 지수 백오프 재시도 | 3회 (5초, 15초, 45초) | "인터넷 연결이 불안정해요" |
| **파일 권한** | `EACCES` / `EPERM` | 관리자 권한 시도 | 1회 | "폴더 위치를 바꿔볼게요" → 바탕화면으로 이동 |
| **디스크 공간** | `ENOSPC` | 복구 불가 | 0회 | "디스크 공간이 부족해요. 불필요한 파일을 삭제해 주세요" |
| **GitHub 인증** | `gh auth` 만료 | `gh auth refresh` | 1회 | "GitHub에 다시 로그인해 주세요" |
| **빌드 에러** | 코드 문법 오류 | Claude Code가 코드 자동 수정 | 3회 | 3회 실패 시 마지막 체크포인트로 롤백 |
| **WebSocket 실패** | 연결 거부 | HTTP 폴링으로 자동 전환 | 전환 1회 | "실시간 연결이 불안정해서 자동 새로고침으로 바꿨어요" |
| **메모리 부족** | Node.js OOM | 복구 불가 | 0회 | "앱이 너무 커졌어요. 기능을 줄여볼까요?" |

**복구 불가 시 최후 수단**:
1. 마지막 Git 체크포인트로 롤백
2. 사역자에게 상황을 한국어 한 문장으로 설명
3. 구체적 대안 제시 (**아래 패턴 참고**)
4. 절대로 영어 에러 메시지, 스택 트레이스, 기술 용어를 노출하지 않음

**구체적 대안 패턴 예시 (3차 성찰 반영)**:

| 실패 상황 | ❌ 나쁜 안내 | ✅ 좋은 안내 |
|-----------|-------------|-------------|
| 실시간 퀴즈 실패 | "기능을 줄여볼까요?" | "실시간 버저 대신, 학생들이 손을 들고 사역자님이 직접 점수를 입력하는 방식으로 바꿔드릴까요? 점수판은 그대로 작동해요." |
| WebSocket 완전 실패 | "연결이 안 돼요" | "실시간 연결이 안 되지만, 5초마다 자동으로 새로고침하는 방식으로 바꿨어요. 살짝 느리지만 똑같이 동작해요." |
| 스탬프 랠리 QR 실패 | "QR이 안 돼요" | "QR 대신 학생들이 미션 장소에서 비밀번호를 입력하는 방식으로 바꿔드릴까요? 비밀번호는 각 장소에 종이로 붙여두시면 돼요." |
| 빌드 에러 3회 | "만들 수 없어요" | "이 기능이 좀 복잡해서 다른 방식으로 만들어볼게요. 결과는 거의 같아요." → 기술 스택 변경 후 재시도 |

### 8.5 Claude Code 도구 레벨 에러 (3차 성찰 반영)

> §8.4는 앱 코드 생성 중의 에러를 다룬다. 이 섹션은 **Claude Code 자체**의 한계로 발생하는 에러를 다룬다.

| 에러 유형 | 원인 | 사역자에게 보이는 현상 | 대응 |
|-----------|------|----------------------|------|
| **컨텍스트 초과** | 대화가 너무 길어짐 (퀴즈 50문제 입력 등) | Claude Code가 이전 대화를 잊음 | "대화가 길어져서 앞부분이 잘 기억이 안 나요. 다시 한번 알려주시겠어요?" + 파일 입력(§6.7.1) 권장 |
| **API 속도 제한** | 짧은 시간에 너무 많은 요청 | 응답이 느려지거나 멈춤 | "잠시 쉬었다가 할게요. 1~2분만 기다려 주세요." |
| **API 연결 끊김** | 인터넷 불안정 | 응답 없음 | "인터넷 연결이 끊어졌어요. 연결되면 알려주세요. 만들던 앱은 안전하게 저장되어 있어요." |
| **코드 생성 중 중단** | 긴 코드 생성 중 타임아웃 | 불완전한 응답 | 마지막 Git 체크포인트에서 재개. "앗, 중간에 끊어졌어요. 이어서 만들게요." |
| **Claude Code 업데이트** | 도구 자체 업데이트 필요 | 실행 불가 또는 경고 | "프로그램을 업데이트해야 해요. 다시 실행하면 자동으로 업데이트돼요." |

### 9.2 품질 검증 (Quality Gate)

> **2차 성찰 반영**: 각 검증 항목에 합격 기준값, 실패 시 동작, 검증 방법을 명시한다.

Claude Code가 앱 생성 후 **자동으로** 실행하는 품질 검증:

| # | 검증 항목 | 합격 기준 | 검증 방법 | 실패 시 동작 |
|---|-----------|-----------|-----------|-------------|
| Q1 | **서버 실행** | localhost에서 HTTP 200 응답 | `curl http://localhost:PORT` 또는 `fetch` | 포트 변경 후 재시도 (§8.4 참고) |
| Q2 | **HTML 유효성** | 렌더링 차단 에러 없음 | 생성된 HTML 구문 검사 | 자동 수정 후 재검증 |
| Q3 | **외부 의존성** | CDN 폰트 외 외부 스크립트 0개 | 생성 코드에서 `<script src="http` 패턴 검색 | 해당 의존성 인라인화 또는 제거 |
| Q4 | **번들 크기** | 300KB 이하 (gzip) | 빌드 산출물 크기 측정 | 이미지 압축 → 코드 분할 → 기능 축소 순으로 최적화 재시도 |
| Q5 | **한국어 렌더링** | 주요 텍스트(제목, 버튼, 메뉴)에 깨짐 없음 | 한글 포함 텍스트 목록 대조 | 폰트 폴백 추가 |
| Q6 | **터치 영역** | 모든 버튼/링크 44×44px 이상 | CSS 속성 검사 | `min-height`, `min-width`, `padding` 자동 보정 |
| Q7 | **QR 코드** | 생성된 QR이 올바른 URL 인코딩 | QR 디코딩 → URL 비교 | QR 재생성 |
| Q8 | **관리자 보호** | `/admin` 접근 시 비밀번호 요구 | 비밀번호 없이 접근 시도 → 차단 확인 | 인증 미들웨어 추가 |
| Q9 | **XSS 방지** | 사용자 입력에 `<script>` 삽입 시 무해화 | 테스트 입력으로 검증 | 이스케이프 함수 추가 |
| Q10 | **실제 화면** | 사역자가 브라우저에서 확인 후 승인 | 사역자 대화 피드백 | 사역자 요청에 따라 수정 |

**Quality Gate 통과 규칙**:
```
Q1~Q9: Claude Code가 자동 실행. 모두 통과해야 사역자에게 미리보기 제공.
Q10:   사역자 확인. 사역자가 "좋아요" / "이대로 해주세요" → 배포 단계 진행.

실패 시: 자동 수정 후 재검증 (최대 3회).
3회 실패 시: 마지막 체크포인트로 롤백 후 사역자에게 안내.
```

### 9.3 사역자 관점 품질

- **대화 → 결과 정확도**: 사역자가 말한 내용이 빠짐없이 반영. 말하지 않은 것은 임의 추가 안 함.
- **이해 가능한 미리보기**: 기술 지식 없이도 수정 요청 가능.
- **롤백 안전망**: "아까 것이 더 좋았어요" → 즉시 복원.

## §10. Claude Code 실행 단계

> **이 섹션은 기존 §10(8주 개발 로드맵)을 대체한다.** 개발팀의 프로젝트 계획이 아니라, Claude Code가 workflow.md를 따라 실행하는 단계별 순서를 정의한다.

### 10.0 상태 전환 조건 & 사용자 확인 신호 (2차 성찰 반영)

> 각 Step 사이의 전환 조건과, "사역자가 확인했다"의 정확한 판단 기준을 정의한다.

**상태 전환 다이어그램**:

```
Step 0 (사전준비)
  │ 전환 조건: Claude Code 실행 + API 키 인증 성공
  ▼
Step 1 (의도파악)
  │ 전환 조건: 사역자 확인 신호 수신 (아래 표 참고)
  ▼
Step 2 (프로젝트 초기화)
  │ 전환 조건: 폴더 생성 + Git init + 패키지 설치 성공 (자동)
  ▼
Step 3 (코드 생성)
  │ 전환 조건: Quality Gate Q1~Q9 전체 통과 (자동)
  ▼
Step 4 (로컬 미리보기)
  │ 전환 조건: 사역자 확인 신호 수신
  ▼
Step 5 (수정 반복)  ←──── 사역자가 수정 요청 시 반복
  │ 전환 조건: 사역자가 "완성" 신호 전달
  ▼
Step 6 (최종 배포)
  │ 완료 조건: URL/QR 생성 + 사역자에게 전달
  ▼
완료
```

**사역자 확인 신호 판단 규칙**:

| 사역자 발화 | 의미 | Claude Code 동작 |
|-------------|------|-------------------|
| "네" / "좋아요" / "그렇게 해주세요" / "시작해주세요" | **승인** → 다음 단계 진행 | 현재 내용 확정, 다음 Step으로 전환 |
| "잠깐" / "잠시만" / "생각 좀 할게요" | **보류** → 기다림 | 대기. 30초 후 "천천히 생각하세요. 준비되면 말씀해 주세요" |
| "이건 아닌데" / "다시" / "바꿔주세요" | **수정 요청** → 현재 단계 반복 | 수정 사항 파악 질문 |
| "뭐가 다른 건지 모르겠어요" | **이해 부족** → 추가 설명 | 기술 없이 결과 차이를 시각적으로 설명 |
| 무응답 (2분 이상) | **이탈 가능** → 리마인드 | "혹시 다른 일 보고 계신가요? 준비되면 말씀해 주세요" |

**확인의 범위 (3차 성찰 반영)**: 사역자 확인은 **결과물의 모습과 기능**(어떤 화면이 있는지, 어떻게 생겼는지)에 대한 것이다. 코드 구조, 보안, 성능 같은 **기술적 타당성은 Claude Code가 자체 판단**한다. 사역자에게 기술적 판단을 전가하지 않는다.

**절대 금지**: 사역자 확인 없이 코드 생성(Step 3)이나 배포(Step 6)를 진행하지 않는다.

### 10.1 Step 0 — 사전 준비 (사역자가 1회만 수행)

이 단계는 Claude Code 실행 전에 사역자가 (또는 도움을 줄 수 있는 사람이) 1회만 수행한다.

**⏰ 권장 준비 시기 (3차 성찰 반영)**: 수련회 **최소 1주일 전**에 Step 0을 완료할 것. Step 0(설치)만 미리 해두면 나머지(앱 생성~배포)는 1~2시간이면 충분하다. 수련회 3일 전에 처음 시작하면 시간 압박으로 좌절할 수 있다.

```
필요한 것:
  1. Windows 10/11 PC
  2. 인터넷 연결
  3. Claude Code 설치 (https://claude.ai/download)
     → 설치 파일을 다운로드하고 실행하면 됨
     → Node.js가 없으면 Claude Code 설치 과정에서 함께 설치됨

환경 사전 점검 (3차 성찰 반영):
  □ 관리자 권한 확인: 프로그램 설치가 가능한 PC인가?
    → 회사 노트북 등 IT부서가 설치를 차단하는 PC는 사용 불가
    → 개인 노트북 또는 교회 전용 PC 권장
  □ 안티바이러스 예외: 알약, V3 등이 Node.js를 차단할 수 있음
    → 차단 시 "이 프로그램을 허용합니다" 선택 안내
  □ 디스크 공간: 최소 2GB 여유 공간 필요
  □ 와이파이 연결 가능한 PC (유선만 되는 데스크톱은 LAN 배포 불가)

Claude Code 첫 실행 — 터미널 안내 (3차 성찰 반영):
  → 검은 화면(터미널)이 뜨면 정상이다.
    "검은 화면이 나오면 놀라지 마세요. 여기에 한국어로
     대화하면 됩니다. 카카오톡처럼 글을 쓰고 엔터를 누르세요."
  → 인증 코드(API 키) 입력 요청이 나타남
    → 사역자가 인증 코드를 붙여넣기
    → 완료. 이후 대화 시작.

Claude Code 허용 설정 (3차 성찰 반영):
  → Claude Code가 명령을 실행할 때 매번 승인을 요청할 수 있음
  → workflow.md에서 안전한 작업(파일 생성, 서버 실행 등)은
    자동 허용되도록 설정을 미리 구성한다
  → 사역자가 매번 "허용" 버튼을 누르지 않아도 되게 함

GitHub Pages 배포를 위한 추가 준비 (선택, 도움 줄 사람이 있을 때만):
  → GitHub 계정 생성 (https://github.com) — 영어 페이지
  → Git 설치 (Claude Code가 자동 안내)
  → GitHub 인증 설정 (Claude Code가 단계별로 안내)
  → ⚠ 이 과정은 기술 도우미(교회 청년 등)와 함께하는 것을 강력 권장
  → 이 과정은 최초 1회만 필요. 이후 모든 배포는 Claude Code가 자동 수행.
  → GitHub Pages가 없어도 LAN 배포로 앱 사용 가능 (필수 아님)

이 과정에서 막히면:
  → Claude Code가 한국어로 안내 메시지를 제공
  → 교회 청년부 중 컴퓨터에 익숙한 사람에게
    "이 프로그램 설치 좀 도와주세요"라고 요청 (1회성)

비용 안내 (투명성 원칙):
  → Claude Code 사용 시 API 비용이 발생한다는 사실을 안내
    (Claude API는 유료 서비스 — §4.5의 "추가 호스팅 비용 제로"와 별개)
  → 앱 1개 생성에 예상되는 대략적 비용 범위 제시
  → GitHub Pages 호스팅은 무료, LAN 배포는 비용 없음
```

### 10.2 Step 1 — 의도 파악 (Claude Code 자동)

```
사역자가 Claude Code에게 원하는 앱을 설명한다.

Claude Code가 수행하는 것:
  - 대화에서 앱 유형 감지 (퀴즈? 일정표? 종합?)
  - 부족한 정보 재질문 (팀 수, 날짜, 콘텐츠 등)
  - 앱 구조를 텍스트로 보여주고 사역자 확인

이 단계의 산출물:
  - 사역자가 확인한 앱 구조 (어떤 화면이 있는지)
  - 콘텐츠 (팀 이름, 퀴즈 문제, 일정 등)
```

### 10.3 Step 2 — 프로젝트 초기화 (Claude Code 자동)

```
Claude Code가 자동으로 수행:
  - 프로젝트 폴더 생성 (영문 경로, 예: C:\church-app\)
  - Git 초기화 (롤백용 체크포인트)
  - 필요한 패키지 설치 (npm install 등)
  - 사역자에게는 "앱을 만들 준비를 하고 있어요" 정도만 표시

사역자가 할 것: 없음 (기다리기)
```

#### Git 체크포인트 전략 (2차 성찰 반영)

> "아까 것이 더 좋았어요"에 대응하기 위한 버전 관리 전략.

**커밋 시점 & 메시지 포맷**:

```
커밋 포맷: "[단계] 설명"

자동 커밋 시점:
  [초기화] 프로젝트 생성 완료
  [콘텐츠] 퀴즈 문제 50개 삽입
  [콘텐츠] 일정표 삽입
  [디자인] 메인 색상 적용
  [디자인] 다크모드 추가
  [기능] 실시간 점수판 구현
  [기능] 관리자 콘솔 추가
  [기능] PWA 오프라인 설정
  [검증] 품질 검증 통과
  [수정] 배경색 파란색 변경     ← 사역자 수정 요청 시
  [수정] 문제 3번 정답 수정     ← 사역자 수정 요청 시
  [배포] 최종 빌드 완료
```

**롤백 처리 방식**:

```
사역자: "아까 것이 더 좋았어요"

Claude Code 내부 처리:
  1. 최근 커밋 목록에서 직전 체크포인트 식별
  2. `git checkout [커밋해시] -- .` 으로 파일 복원
  3. 새 커밋으로 기록: "[롤백] 이전 디자인으로 복원"
  4. 브라우저 새로고침

사역자: "처음 것으로 돌아가주세요"

Claude Code:
  1. 태그 기반 주요 지점 검색
  2. 가장 이른 의미 있는 체크포인트로 복원
  3. "처음 만들었을 때의 모습으로 돌려놨어요. 확인해 보세요."

보관 정책:
  - 모든 커밋을 보관 (삭제하지 않음)
  - Git 히스토리는 사역자에게 노출하지 않음
  - Claude Code가 내부적으로만 관리
```

### 10.4 Step 3 — 코드 생성 (Claude Code 자동)

```
Claude Code가 직접 작성하는 것:
  - 앱 전체 코드 (프론트엔드 + 백엔드)
  - 스타일링 (색상, 폰트, 레이아웃)
  - 콘텐츠 삽입 (퀴즈 문제, 일정, 성경 구절 등)
  - 실시간 기능 (필요 시 WebSocket + 폴링 폴백)
  - PWA 설정 (오프라인 캐싱)
  - 관리자 콘솔 (/admin)

Git 체크포인트: 주요 단계마다 자동 커밋

사역자에게 보여주는 진행 상황 (3차 성찰 반영):
  → 코드 생성 중 사역자 화면에 영어 코드가 쏟아진다.
    사역자가 불안해하지 않도록 중간중간 한국어 안내를 삽입:
  → "지금 앱의 뼈대를 만들고 있어요 (1/5 단계)"
  → "디자인을 입히고 있어요 (2/5 단계)"
  → "퀴즈 문제를 넣고 있어요 (3/5 단계)"
  → "실시간 기능을 추가하고 있어요 (4/5 단계)"
  → "마무리 점검 중이에요 (5/5 단계)"
  → 각 단계 사이에 "잘 되고 있어요. 조금만 기다려 주세요." 안내
```

### 10.5 Step 4 — 로컬 미리보기 (Claude Code + 사역자)

```
Claude Code가 자동으로:
  - 로컬 서버 실행
  - 브라우저 자동 열기
  - 같은 와이파이 접속용 QR코드 표시

사역자가 할 것:
  - PC 브라우저에서 확인
  - 스마트폰으로 QR 스캔하여 모바일에서도 확인
  - 수정할 점을 Claude Code에게 대화로 전달
```

### 10.6 Step 5 — 수정 반복 (Claude Code + 사역자)

```
사역자: "팀 색상을 바꿔주세요" / "문제 3번을 수정해주세요" / "글씨가 작아요"
Claude Code: 즉시 코드 수정 → 브라우저 자동 새로고침
사역자: "아까 것이 더 좋았어요"
Claude Code: Git 롤백 → 이전 버전으로 복원

이 단계는 사역자가 만족할 때까지 반복한다.
```

### 10.7 Step 6 — 최종 배포 (Claude Code 자동)

```
Claude Code가 앱 유형에 따라 자동으로 배포 경로를 선택한다:

[정적 앱인 경우 — GitHub Pages 배포]
  1. 프로덕션 빌드 (최적화, 압축)
  2. 품질 검증 (§9.2 항목 자동 확인)
  3. GitHub 저장소 생성 (또는 기존 저장소에 푸시)
  4. GitHub Pages 자동 활성화
  5. 배포 완료 → 공개 URL 생성
  6. URL 기반 QR코드 자동 생성
  7. 인쇄용 PDF 생성 (A4, 수련회 이름 + QR + 안내문)

  사역자에게: "앱이 완성됐어요! 이 URL을 학생들에게 공유하세요.
              QR코드 PDF도 만들었어요. 인쇄해서 수련회 장소에 붙여주세요."
              → URL은 사역자 PC가 꺼져 있어도 접속 가능

[실시간 앱인 경우 — LAN 서버 배포]
  1. 프로덕션 빌드 (최적화, 압축)
  2. 품질 검증 (§9.2 항목 자동 확인)
  3. 로컬 서버 실행
  4. 같은 와이파이 접속용 QR코드 생성
  5. 인쇄용 PDF 생성

  사역자에게: "앱이 완성됐어요! 수련회 때 노트북을 켜고
              '앱 실행해줘'라고 말씀해 주세요.
              QR코드 PDF를 인쇄해서 수련회 장소에 붙여주세요."

[공통]
  - 수련회 현장 운영 안내 (§11 참고)
  - Git 체크포인트로 언제든 이전 상태 복원 가능
```

### 10.8 세션 중단 & 재개 (2차 성찰 반영)

> 사역자가 Claude Code를 닫거나, PC를 끄거나, 작업 도중 다음 날 이어서 하려는 경우.

**중단 시나리오와 대응**:

| 시나리오 | 프로젝트 상태 | 재개 방법 |
|----------|--------------|-----------|
| **사역자가 Claude Code 창을 닫음** | 코드는 프로젝트 폴더에 그대로 보존. Git 커밋도 유지. 서버만 중단됨 | Claude Code 재실행 → 프로젝트 폴더에서 시작 → "이전에 만들던 앱이 있어요. 이어서 할까요?" |
| **PC 재부팅** | 위와 동일 | 위와 동일 |
| **작업 도중 내일 이어서** | 위와 동일 | 위와 동일 |
| **npm install 중간에 중단** | 불완전한 node_modules | Claude Code가 `node_modules` 삭제 후 재설치 자동 수행 |
| **코드 생성 중간에 중단** | 불완전한 코드 상태 | 마지막 Git 체크포인트에서 재개. 불완전 코드는 롤백 |

**재개 시 Claude Code 행동**:

```
Claude Code 재실행 시:

[1] 프로젝트 폴더 존재 확인
    → 존재하면: "이전에 만들던 [앱 이름]이 있어요. 이어서 할까요?"
    → 없으면: 새 프로젝트 시작

[2] Git 히스토리에서 마지막 상태 파악
    → 마지막 커밋 메시지로 중단된 단계 식별
    → "[디자인] 메인 색상 적용" → Step 3 (코드 생성) 중간

[3] 해당 단계부터 재개
    → "지난번에 디자인까지 했어요. 이어서 기능을 만들까요?"

[4] 서버 재시작 (미리보기가 필요한 경우)
    → 이전과 동일한 포트로 서버 실행
    → QR코드 재생성 (IP가 바뀌었을 수 있으므로)
```

**핵심 원칙**: 코드와 콘텐츠는 **프로젝트 폴더와 Git에 영구 저장**되므로, Claude Code 세션이 종료되어도 작업물은 절대 손실되지 않는다. Claude Code는 세션 상태가 아닌 **파일 시스템과 Git 히스토리**를 기준으로 작업 상태를 복원한다.

## §11. 현장 운영 & 장애 대응

> **이 섹션은 수련회 현장(2박 3일)에서의 실제 운영 시나리오를 다룬다.**

### 11.1 수련회 현장 체크리스트

사역자가 수련회 장소에 도착한 후:

```
0. 와이파이 수용 인원 확인 (3차 성찰 반영)
   → 공유기의 최대 동시 접속 수 확인 (일반 가정용: 20~30대)
   → 학생 35명 이상이면: 공유기 2대 사용 또는 팀별 시간차 접속 고려
   → ⚠ 모바일 핫스팟 폴백 시: 최대 약 8대만 동시 접속 가능
     (팀별로 나눠서 접속하거나, 한 팀씩 돌아가며 사용)
1. 노트북을 전원에 연결 (배터리 모드 금지 — 절전 방지)
2. 수련회 장소 와이파이에 연결
3. Claude Code에서 "앱 실행해줘"라고 말함
4. Claude Code가 서버 시작 + QR코드 표시
5. QR코드를 인쇄하거나, 빔프로젝터에 표시
6. 학생들이 스마트폰으로 QR 스캔 → 접속
```

### 11.2 사역자 PC = 서버라는 사실

실시간 기능(퀴즈, 점수판, 가사 동기화)이 있는 앱은 사역자 PC가 서버 역할을 한다.

| 상황 | 영향 | 대응 |
|------|------|------|
| **노트북 절전 모드** | 서버 중단 → 학생 접속 불가 | Windows 전원 설정을 "절전 안 함"으로 변경 (Claude Code가 자동 안내) |
| **노트북 덮개 닫기** | 절전 → 서버 중단 | "노트북을 펼쳐둔 채로 유지해 주세요" 안내 |
| **와이파이 끊김** | 학생 접속 불가 | 모바일 핫스팟으로 전환 안내 |
| **정전** | 모든 것 중단 | 데이터는 SQLite에 저장되어 있으므로, 전원 복구 후 서버 재시작하면 복원 |
| **서버 중단 복구** | 학생 화면에 에러 표시 | **목표: 30초 이내 복구.** Claude Code에서 "앱 실행해줘" → 서버 재시작 → 학생 화면 자동 재연결. 학생에게는 "연결 대기 중..." 표시 후 자동 복구 |

### 11.3 와이파이 문제 대응

```
[문제] 학생 폰에서 앱이 안 열림

Claude Code 안내:
  "혹시 학생들 폰이 와이파이에 연결되어 있는지 확인해 주세요.
   그래도 안 되면, 노트북에서 모바일 핫스팟을 켜고
   학생들이 그 핫스팟에 연결하면 돼요.
   제가 새 QR코드를 만들어 드릴게요."

[문제] 와이파이가 느려서 로딩이 오래 걸림

대응: PWA 캐싱 덕분에 최초 1회만 느림.
      한 번 접속한 학생은 캐시에서 즉시 로딩.
      "처음 접속할 때 좀 기다려야 해요. 한 번만 기다리면 그 다음부터는 빨라요."
```

### 11.4 실시간 기능 장애 대응

```
[문제] 퀴즈 점수가 실시간으로 안 바뀜

원인: WebSocket 연결 실패 (방화벽, 클라이언트 격리 등)
자동 대응: 5초 폴링으로 자동 전환 (§6.3)
사역자 안내: "점수가 5초마다 자동으로 새로고침돼요. 조금 느리지만 정상이에요."

[문제] 완전히 연결 안 됨 (오프라인)

정적 콘텐츠(일정, QT, 가사): PWA 캐시에서 정상 동작
실시간 콘텐츠(점수판, 퀴즈): "연결 대기 중..." 표시, 연결 복구 시 자동 반영
```

### 11.5 수련회 종료 후

```
Claude Code에게: "수련회 끝났어요"

Claude Code:
  "수고하셨어요! 앱 데이터는 노트북에 저장되어 있어요.
   나중에 삭제하고 싶으시면 [폴더 경로]를 삭제하시면 돼요.
   서버를 종료할게요."
```

## §12. 명시적 제외 항목

아래 항목들은 **의도적으로 제외**한다. 핵심 목표 달성을 위한 전략적 선택.

| 제외 항목 | 제외 이유 | 대안 |
|-----------|-----------|------|
| **별도 "앱 빌더" 시스템 구축** | Claude Code가 직접 코드를 생성하므로 중간 도구 불필요 | Claude Code 자체가 빌더 |
| **네이티브 앱 (iOS/Android)** | 스토어 심사 불필요, PWA로 충분 | PWA (브라우저 기반) |
| **다국어 지원** | 대상이 한국 교회로 한정 | 한국어 최적화 집중 |
| **결제/후원 기능** | 법적 요건, 보안 복잡도 | 기존 수납 방식 |
| **사용자 계정 시스템** | 개인정보 수집 최소화, 운영 리스크 | 닉네임 + 세션 쿠키 |
| **AI 챗봇 (학생 대상)** | API 비용 폭증, 콘텐츠 안전 | AI는 앱 생성에만 사용 |
| **영상 스트리밍** | CDN 비용, 저작권 | YouTube/Zoom 링크 |
| **Ollama 로컬 LLM 폴백** | 사역자 PC 사양 부족 (GPU 필요), 코드 생성 품질 저하 | Claude API 불가 시 솔직히 안내 |
| **성경 전체 DB 자동 구축** | 저작권 미확인 (§4.8) | 사역자 직접 입력 또는 AI 생성 |
| **학생 → 서버 사진 업로드** | 부적절 이미지 위험, 용량 관리 | 사역자만 업로드 (단방향) |
| **50명 자동 부하 테스트** | 사역자 PC에서 비현실적, 결과 왜곡 가능 | 실제 현장에서 점진적 확인 |
| **SaaS 배포 / 유료 클라우드 호스팅** | 비용 제로 원칙 (§4.5). GitHub Pages(무료)와 LAN 배포로 충분 | GitHub Pages(정적) + LAN(실시간) |
| **8주 개발 로드맵** | 개발팀이 없음. Claude Code가 실시간 생성 | §10 실행 단계로 대체 |
| **별도 CLI (camp-builder)** | Claude Code 자체가 대화 인터페이스 | Claude Code 직접 대화 |

```
│  ❌ 학생 대상 AI 챗봇 / 영상 스트리밍          │
│  ❌ SaaS 배포 / 클라우드 호스팅                │
│  ❌ 별도 CLI 도구 (camp-builder)               │
└─────────────────────────────────────────────────┘
```

## §13. 위험 가정 & 검증

> **성찰 반영**: 이 프로젝트의 성공을 좌우하는 핵심 가정들을 명시하고, 각각의 검증 방법을 정의한다. 이 가정들이 틀릴 경우의 대응책도 포함한다.

| 순위 | 가정 | 왜 위험한가 | 검증 방법 | 대응책 |
|------|------|-------------|-----------|--------|
| **1** | **사역자가 Claude Code 설치 과정을 완료할 수 있다** | Node.js 설치, API 키 설정, GitHub 계정 생성 등 기술적 장벽이 존재. 여기서 포기하면 이후 모든 단계가 무의미 | 실제 사역자 1명에게 문서만 보고 설치하게 하기. 막히는 지점 기록 | 설치 과정을 영상으로 제작. 교회 청년부 도우미 매칭. 최악의 경우 원격 지원 |
| **2** | **한국어 대화만으로 앱 요구사항이 충분히 구체화된다** | 사역자가 원하는 것을 말로 설명하지 못하거나, Claude Code가 의도를 오해할 수 있음 | 시나리오 3.2~3.5를 실제로 실행하고, 사역자 의도와 결과물의 일치도 확인 | 시각적 예시(스크린샷)를 더 많이 제시. 선택지를 구체적 이미지로 보여주기 |
| **3** | **GitHub Pages 배포까지 자동화가 가능하다** | Git/GitHub 설정, 인증, push, Pages 활성화 등을 Claude Code가 대화만으로 안내하고 자동 실행해야 함 | Claude Code에서 실제로 GitHub Pages 배포 전 과정을 자동화 테스트 | GitHub Pages가 어려우면 Cloudflare Pages 등 대안 검토. 최후 수단: LAN 전용 |
| **4** | **중학생이 완성된 앱을 "진짜 앱"으로 인식한다** | 디자인 품질, 반응 속도, 인터랙션이 인스타/카카오 수준에 미달하면 "촌스럽다"고 외면 | 실제 중학생 3~5명에게 완성된 앱을 보여주고 반응 관찰 | 디자인 템플릿 강화. 애니메이션/전환 효과 추가. 중학생 피드백 기반 반복 수정 |
| **5** | **교회 와이파이 환경에서 앱이 안정적으로 작동한다** | 저대역폭, 클라이언트 격리, 방화벽 등 예측 불가 네트워크 환경 | 실제 교회/수련회장에서 50명 동시 접속 테스트 | PWA 오프라인 캐싱 강화. 모바일 핫스팟 폴백. 정적 앱(GitHub Pages) 우선 권장 |
| **6** | **사역자가 혼자서 두 번째 앱도 만들 수 있다** | 첫 번째는 도움을 받더라도, 두 번째부터 독립적으로 할 수 있어야 진정한 성공 | 사역자가 첫 앱 완성 후 2주 뒤 다른 앱을 혼자 만들어보게 하기 | 사용 가이드 보강. 자주 묻는 질문(FAQ) 정리. "지난번처럼 만들어줘" 패턴 지원 |

### 13.1 가정 검증 우선순위

```
가정 1 (설치) → 가정 2 (대화) → 가정 3 (배포)
  이 3개가 통과하지 못하면 나머지는 의미 없음.
  순서대로 검증하고, 하나라도 실패하면 해당 단계를 해결한 후 다음으로 진행.

가정 4 (디자인) + 가정 5 (네트워크)
  앱이 만들어진 후 현장에서 검증.

가정 6 (반복 사용)
  최소 1회 성공 경험 후 검증.
```

### 13.2 "성공"과 "실패"의 정의

**성공**: 코딩 경험 제로인 사역자가, 도움 없이, Claude Code와 한국어 대화만으로, 중학생이 실제로 사용하는 앱을 완성하고 수련회에서 운영한다.

**부분 성공**: 최초 설치에는 도움이 필요했지만, 이후 앱 생성/수정/배포는 사역자가 독립적으로 수행한다.

**실패**: 사역자가 기술적 장벽 때문에 중간에 포기한다. 또는 완성된 앱을 중학생이 사용하지 않는다.
