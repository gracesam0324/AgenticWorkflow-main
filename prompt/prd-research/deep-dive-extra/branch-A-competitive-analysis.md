## 1. Open Interpreter

### 핵심 기능 및 작동 방식

핵심 루프: 사용자 메시지 → LLM이 코드 블록 생성 → 사용자 확인(또는 자동 승인) → 로컬 실행 → 결과를 LLM에 피드백 → 반복.

```python
# Open Interpreter의 핵심 실행 루프 (단순화)
class OpenInterpreter:
    def chat(self, message):
        # 1. LLM에 메시지 전송
        # 2. 스트리밍 응답에서 코드 블록 파싱
        # 3. Computer 객체를 통한 코드 실행
        # 4. 결과를 대화 히스토리에 추가
```

`Computer` 객체가 핵심 추상화 레이어. `computer.run(language, code)` 인터페이스로 실제 로컬 환경 코드 실행. subprocess 또는 IPython 커널 사용.

### 주요 약점 (실제 GitHub Issues 기반)

1. **안전성 문제**: 기본 설정에서 `--auto_run` 플래그 시 확인 없이 코드 실행. `rm -rf` 수준 명령 실행 사례 보고.
2. **상태 비유지 문제**: 세션 간 컨텍스트 저장 불안정. 대화 히스토리가 LLM 컨텍스트 창 초과 시 앞부분 잘라냄.
3. **복잡한 멀티스텝 워크플로우 취약**: 단일 대화 흐름 의존. 조건부 분기 로직 없음. 에러 발생 시 LLM이 즉흥 대응하여 재현성 낮음.
4. **버전 불안정성**: 0.x 버전대에서 major breaking changes 잦음.
5. **의존성 충돌**: Jupyter/IPython 커널 의존성으로 환경 충돌 이슈 다수.

### 우리 제품과의 차별점

| 항목 | Open Interpreter | 우리 제품 |
|------|-----------------|----------|
| 실행 방식 | 즉흥적 코드 생성 + 실행 | 사전 정의 YAML 플랜 → 단계별 실행 |
| 재현성 | 낮음 (LLM 출력이 매번 다름) | 높음 (YAML 플랜이 버전 관리됨) |
| 감사 추적 | 없음 | YAML + Git으로 모든 실행 기록 |
| 안전 제어 | 수동 확인 또는 자동 실행 | 플랜 단계별 명시적 승인 게이트 |
| 타겟 사용자 | 일반 사용자 | 보안 연구자, 데이터 엔지니어 |

**핵심 차이**: Open Interpreter는 "지금 당장 뭔가를 해줘" 방식, 우리 제품은 "계획을 세우고, 검토하고, 실행해" 방식.

## 2. AutoGPT / AgentGPT

### 왜 메인스트림이 되지 못했는가 (실패 요인)

1. **무한 루프 문제**: 에이전트가 목표 미달성 상태에서 동일 서브태스크 반복. "AutoGPT가 $50 API 비용을 하룻밤에 소모" Reddit 게시물 수백 개.
2. **작업 완료 판단 실패**: "완료"를 LLM이 결정하는 구조 → 목표 모호 시 무한 실행.
3. **실제 유용성 부재**: 복잡한 멀티스텝에서 hallucination 누적 → "인상적인 데모, 실용성 없음" 패턴.
4. **비용 예측 불가능성**: API 호출 횟수 폭발적 증가. 사용자 통제 불가.
5. **아키텍처적 한계**: 벡터 메모리(Pinecone 등) 의존 → 관련성 없는 메모리가 컨텍스트 오염.

AutoGPT 실패에서 우리 제품 핵심 원칙 도출:
- **명시적 플랜**: 자연어 → YAML 플랜이 사용자에게 먼저 보여지고 승인됨. "에이전트가 알아서 하는" 방식 거부.
- **결정론적 실행**: 동일한 YAML 플랜은 동일하게 실행됨. LLM은 플랜 생성 단계에서만 관여.
- **비용 예측 가능**: API 호출 횟수가 플랜의 단계 수에 비례하여 제한됨.
- **중단점 설계**: 각 YAML 단계가 명시적 체크포인트. 언제든 중단, 재개, 수정 가능.

## 3. n8n / Zapier / Make.com

### 로컬 실행 vs 클라우드 기반의 실질적 차이

n8n이 셀프호스팅을 지원해도, "서버에 배포된 n8n"은 "로컬 컴퓨터에서 직접 실행"과 다르다. 보안 연구자가 분석 중인 멀웨어 샘플, 데이터 엔지니어의 PII 데이터, 학술 연구자의 미공개 데이터셋은 어떤 외부 서버도 경유해선 안 된다. **우리 제품의 로컬 실행은 아키텍처적 선택이 아니라 보안 약속이다.**

### 비기술 사용자 vs 개발자 타겟의 차이

Zapier/Make.com의 GUI 기반 접근법 한계:
- 복잡한 조건 분기를 GUI로 표현하면 스파게티가 됨
- 버전 관리 불가능 (워크플로우를 Git에 커밋할 수 없음)
- 재사용 가능한 컴포넌트 추상화 어려움

YAML 기반 플랜은 개발자에게 자연스럽다. 텍스트 파일이기 때문에 Git 버전 관리, diff 비교, CI/CD 파이프라인 연동 가능.

## 4. LangChain / LlamaIndex 기반 도구들

### 프레임워크 기반 에이전트의 공통 문제점

2024년부터 "LangChain fatigue" 현상:

1. **추상화 오버헤드**: Chain, Agent, Tool, Memory 추상화 계층이 너무 많아 디버깅 어려움. "LangChain을 쓰면 LangChain 디버깅을 하게 된다" — HackerNews 반복 등장.
2. **버전 불안정성**: `langchain-community`, `langchain-core`, `langchain` 패키지 분리 이후 의존성 관리 복잡.
3. **의존성 지옥**: `pip install langchain`이 50개 이상 패키지 설치. 다수가 하위 호환성 없는 업데이트.
4. **성능 오버헤드**: 단순한 LLM 호출에도 여러 미들웨어 레이어 통과.

### 우리가 직접 구현하는 것이 유리한 이유

1. **Anthropic SDK는 안정적**: Messages API는 하위 호환성 유지.
2. **YAML 플랜이 추상화 레이어를 대체**: 프레임워크 없이도 구조화된 실행 가능.
3. **디버깅 용이성**: LLM 호출 횟수, 입출력, 실행 순서가 YAML 플랜에서 명시적으로 드러남. 블랙박스 없음.
4. **배포 단순성**: 의존성이 `anthropic`, `pyyaml`, `gitpython` 정도로 최소화.

## 5. Claude Code / Copilot Workspace

### AI 코딩 어시스턴트와의 포지셔닝 차이

- **Claude Code**: 소프트웨어 개발 작업 (함수 작성, 버그 수정, 리팩토링). 코드 작성이 목적.
- **Copilot Workspace**: GitHub 이슈 → 코드 변경 파이프라인. 소프트웨어 개발 라이프사이클 특화.
- **우리 제품**: 로컬 컴퓨터에서 도메인별 전문 작업 자동화 (데이터/보안/연구). 코드는 수단, 분석/자동화가 목적.
