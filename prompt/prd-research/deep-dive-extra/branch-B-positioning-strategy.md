**경쟁 공백 (Blue Ocean)**: "로컬 + 개발자/전문가 특화" 사분면이 현재 가장 취약.

- Open Interpreter: 로컬이지만 전문가 특화 아님 (범용 코드 실행 도구)
- Claude Code: 개발자 특화이지만 로컬 아닌 API 의존 + 코딩 도구
- n8n 셀프호스팅: 로컬에 가깝지만 GUI 기반으로 개발자보다 운영자 타겟

**우리 제품이 채울 공간**: "로컬 + 도메인 전문가 (데이터/보안/연구) 특화"

## 2. 핵심 차별화 포인트

### 기술적 차별점 3가지

**① YAML 플랜의 감사 추적성 (Auditability)**
경쟁 도구들은 실행 과정이 블랙박스이거나 로그로만 남는다. YAML 플랜은 사용자가 검토하고 승인하는 명시적 계획서다. Git에 커밋되면 "언제, 누가, 어떤 플랜으로, 어떤 결과를 얻었는지"가 완전히 재구성 가능하다. 보안 연구자에게 이것은 직업적 필수 요건이다.

**② LLM 호출의 결정론적 분리**
AutoGPT 계열은 "실행 중에도 LLM이 다음 행동을 즉흥적으로 결정"하는 구조다. 우리 제품은 LLM 관여를 플랜 생성 단계로 격리한다. 일단 YAML 플랜이 승인되면, 실행 단계에서 LLM은 개입하지 않는다. 이것이 재현성과 비용 예측 가능성을 동시에 확보한다.

**③ 플랜의 재사용성과 파라미터화**
YAML 플랜은 텍스트 파일이므로 파라미터 주입 가능. 동일한 "데이터 전처리 플랜"을 다른 입력 파일에 반복 적용하거나, 플랜 라이브러리 구축하여 팀 내 공유 가능. CLI 스크립트와 유사한 재사용 패턴이지만 자연어로 생성되고 인간이 읽기 쉽다.

### UX/경험적 차별점 3가지

**① "놀라움 없는 실행 (No Surprise Execution)"**
"무엇을 실행할 것인지"를 사용자가 미리 알 수 있는 구조. YAML 플랜이 중간 확인점 역할. 보안 의식이 높은 사용자에게 심리적 안전감 제공.

**② 터미널-네이티브 경험**
타겟 사용자(데이터 엔지니어, 보안 연구자, 학술 연구자)는 모두 터미널을 일상 사용. GUI 워크플로우 에디터보다 CLI + YAML이 기존 워크플로우와 자연스럽게 통합된다.

**③ 점진적 자동화 (Progressive Automation)**
처음에는 모든 단계에서 확인 요청 → 사용자가 특정 플랜 패턴에 신뢰를 쌓으면 자동 실행 허용. SSH authorized_keys, sudo 타임아웃과 유사한 철학.

### 철학적 차별점 2가지

**① 로컬 우선이 기능이 아닌 제약 조건**
많은 도구들이 "옵션으로 로컬 실행 지원"을 내세운다. 우리 제품은 클라우드 실행 자체가 아키텍처에 존재하지 않는다. "우리는 로컬 외의 옵션을 고려하지 않았다"는 것이 신뢰의 근거가 된다.

**② 사용자가 도구를 소유 (User Ownership)**
SaaS 도구는 서비스 종료, 요금제 변경, 정책 변경으로 사용자 워크플로우를 인질로 잡을 수 있다. YAML 플랜과 로컬 실행은 완전히 사용자 소유다. 우리 서비스가 종료되어도 YAML 플랜은 남는다.

## 3. 경쟁 제품에서 배울 점

### Open Interpreter에서 배울 것
- **대화형 플랜 수정**: YAML 플랜 생성 후 "3번 단계를 수정해줘" 자연어 명령으로 플랜 수정 인터페이스. 사전 계획의 안전성 + 대화형 편의성 동시 확보.
- **멀티 런타임 지원 패턴**: YAML 스텝이 "어떤 실행기(executor)로 실행될지"를 명시하는 구조.

### n8n에서 배울 것
- **실행 히스토리**: `plan history` 명령어로 과거 실행 결과와 플랜 조회. Git log와 통합하면 강력해짐.
- **에러 처리 패턴**: YAML 플랜의 `on_failure` 섹션으로 구현 가능.

### Claude Code에서 배울 것
- **도구 호출의 명시적 표시**: 각 YAML 단계 실행 전 "이 단계에서 실행될 명령: ..."을 명확하게 표시.
- **컨텍스트 인식**: 자연어 입력 처리 전에 현재 디렉토리 구조, 최근 실행 이력, 환경 변수 등을 자동으로 컨텍스트에 포함.

## 4. 경쟁 우위 지속 방법

### 로컬/프라이버시 니치 고수
2024-2025년 AI 도구 시장의 명확한 추세 중 하나는 "프라이버시 반발". ChatGPT 데이터 학습 정책, Copilot 코드 유출 우려, 기업들의 AI 도구 사용 금지 정책이 이를 반증한다. 대형 SaaS 기업은 구조적으로 "로컬 전용 도구"를 만들 수 없다. 비즈니스 모델이 클라우드 트래픽에 기반하기 때문이다. 이것이 1인 개발자의 구조적 우위다.

### 커뮤니티 기반 성장 전략
- **데이터 엔지니어**: dbt Slack, Apache Airflow 커뮤니티, DataEng Subreddit
- **보안 연구자**: DEFCON 커뮤니티, 보안 관련 Discord/Slack, 버그 바운티 플랫폼
- **학술 연구자**: 각 분야 arXiv 관련 Discord, ResearchGate, 학술 컨퍼런스 워크샵

"로컬 LLM으로 100GB 로그 분석 파이프라인 구축기" 같은 구체적 사용 사례가 기술 커뮤니티에서 자연스럽게 확산된다.

## 5. PRD에 반영해야 할 경쟁 인텔리전스

### 피해야 할 실수 (경쟁 도구가 망한 이유)

- **AutoGPT로부터**: "에이전트의 자율성보다 사용자의 통제권이 항상 우선한다."
- **Open Interpreter로부터**: 안전 기본값의 중요성. 기본값은 "모든 단계에서 확인 요청"이어야 하며, 자동 실행은 명시적 사용자 설정이 필요하다.
- **LangChain으로부터**: "YAML 플랜의 각 단계는 단독으로 이해 가능해야 하며, 실행 로그는 사람이 읽을 수 있는 형태로 보존된다."
- **n8n/Zapier로부터**: GUI 워크플로우 에디터 유혹을 피하라. CLI + YAML에 집중.

### 반드시 해결해야 할 사용자 불만 (경쟁 도구 실패에서 도출)

1. **비용 예측 불가능성**: API 비용이 얼마나 발생할지 플랜 생성 전에 대략적으로 알 수 있어야 함.
2. **실패 복구의 어려움**: 10단계 중 7단계에서 실패했을 때 체크포인트 시스템이 MVP에서도 반드시 포함되어야 함.
3. **환경 재현성**: 특정 CLI 도구나 Python 패키지가 없을 때 명확한 에러 메시지와 설치 가이드.
4. **플랜 디버깅 어려움**: 각 단계의 입출력을 verbose 모드에서 확인 가능해야 함.

### 포지셔닝 선언문

> "보안을 타협하지 않는 데이터 엔지니어와 연구자를 위한, 로컬 컴퓨터에서 완전히 실행되는 AI 워크플로우 자동화 도구. 당신의 데이터는 당신의 컴퓨터를 떠나지 않는다."

> "자연어로 계획하고, YAML로 검토하고, 로컬에서 실행한다. AutoGPT의 자율성이 아니라 Git의 재현성으로 AI 워크플로우를 관리하는 도구."

## 종합 경쟁 인텔리전스 요약

| 경쟁 도구 | 핵심 실패/한계 | 우리의 대응 |
|-----------|--------------|------------|
| AutoGPT | 비용 폭주, 무한루프, 제어 불가 | YAML 플랜으로 실행 범위 명시적 제한 |
| Open Interpreter | 안전 기본값 부재, 재현성 없음 | 확인 우선 기본값, YAML 버전 관리 |
| n8n/Zapier | 데이터 외부 유출, GUI 한계 | 완전 로컬, CLI + YAML |
| LangChain | 의존성 지옥, 디버깅 난해 | 최소 의존성, 명시적 실행 흐름 |
| Claude Code | 코딩 도구 포지셔닝, 클라우드 의존 | 도메인 전문가 특화, 완전 로컬 |

현재 시장에서 "로컬 + 전문가 + 재현 가능한 워크플로우"를 동시에 충족하는 도구는 존재하지 않는다.
